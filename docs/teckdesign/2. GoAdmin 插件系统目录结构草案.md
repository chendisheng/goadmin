# GoAdmin 插件系统目录结构草案 v1.0

> **定位**：面向落地实现的插件系统目录结构草案。
>
> **目标**：把插件系统从“概念设计”拆成“可直接开发”的目录边界、职责分工和演进路径。
>
> **约束**：严格遵循 `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`，并与当前已实现的插件 CRUD、registry、loader、builtin 插件保持一致。

---

# 1. 草案目标

本草案只回答一件事：**插件系统代码应该怎么分目录，才能方便开发、测试、扩展和治理。**

它不再强调高层概念，而是直接给出：

- 目录应该怎么拆
- 每个目录负责什么
- 哪些能力放在哪个子域
- 现有代码如何平滑演进过去

---

# 2. 设计原则

## 2.1 先分边界，再分文件

插件系统最容易出问题的地方不是“没有功能”，而是“职责混在一起”。

因此目录划分必须优先保证：

- 清单与发现分离
- 生命周期与运行态分离
- 市场化与本地加载分离
- 业务注册与基础设施分离

## 2.2 保持可渐进演进

现有代码已经具备：

- `interface/`
- `loader/`
- `registry/`
- `application/`
- `domain/`
- `infrastructure/`
- `transport/`
- `builtin/`

所以草案不能推翻重来，而应在现有结构上补齐：

- `manifest/`
- `discovery/`
- `lifecycle/`
- `marketplace/`
- `runtime/`

---

# 3. 推荐目录总览

```text
backend/plugin/
  interface/          # 插件契约、上下文、注册器接口
  manifest/            # 插件清单、schema、版本与依赖描述
  discovery/           # 插件扫描、索引、解析与候选构建
  lifecycle/           # 启停、启用、禁用、升级、回滚编排
  marketplace/         # 插件市场、下载、安装、分发
  runtime/             # 运行态装配、资源挂载、实例管理
  loader/              # 插件装载器
  registry/            # 路由/菜单/权限/事件注册中心
  builtin/             # 内置插件
  application/         # 插件用例编排、服务层
  domain/              # 插件领域模型、仓储接口
  infrastructure/      # 仓储实现、外部适配、持久化
  transport/           # HTTP/CLI/任务调度等入口
```

# 4. 各目录职责

## 4.1 `interface/`

职责：定义插件系统最小契约。

建议放置：

- `Plugin` 接口
- `Registrar` 接口
- `Context` 结构
- `PluginMetadata` 结构
- `Route` / `Menu` / `Permission` 基础定义

这个目录只做“协议定义”，不做业务逻辑。

## 4.2 `manifest/`

职责：描述插件是什么。

建议放置：

- manifest 结构体
- schema 校验
- 版本语义解析
- 依赖描述
- 能力声明
- 签名与校验信息

建议文件示例：

```text
manifest/
  manifest.go
  schema.go
  parser.go
  validator.go
```

## 4.3 `discovery/`

职责：发现插件在哪里、是否可用。

建议放置：

- 本地目录扫描
- builtin 插件索引
- marketplace 包索引
- manifest 读取
- 依赖图构建
- 兼容性预检查

建议文件示例：

```text
discovery/
  scanner.go
  resolver.go
  index.go
  planner.go
```

## 4.4 `lifecycle/`

职责：统一管理插件状态流转。

建议放置：

- 状态机
- 启用 / 禁用
- 加载 / 初始化 / 注册 / 启动 / 停止
- 升级 / 回滚
- 热切换编排

建议文件示例：

```text
lifecycle/
  state.go
  manager.go
  transition.go
  rollback.go
```

## 4.5 `marketplace/`

职责：管理插件包的安装与分发。

建议放置：

- 市场索引查询
- 下载包
- 安装 / 卸载 / 升级
- 签名校验
- 兼容性检查
- 包缓存管理

建议文件示例：

```text
marketplace/
  client.go
  catalog.go
  installer.go
  package_store.go
```

## 4.6 `runtime/`

职责：把插件真正挂到运行时。

建议放置：

- 插件实例装配
- 依赖注入
- 运行时资源挂载
- 插件健康检查
- 热更新后的实例切换

建议文件示例：

```text
runtime/
  runtime.go
  instance.go
  binder.go
  health.go
```

## 4.7 `loader/`

职责：按顺序装载插件实例。

当前代码中的 loader 已经承担基础加载职责，后续应尽量保持轻量，只负责：

- 排序
- 调用 `Register`
- 组合注册结果
- 将装载动作交给 `runtime/` 或 `lifecycle/`

## 4.8 `registry/`

职责：收敛插件对外注册的资源。

已存在能力应继续保留：

- 路由注册
- 菜单注册
- 权限注册
- 事件注册
- 任务注册

建议 registry 只做“收集”和“查询”，不要承载生命周期控制。

## 4.9 `builtin/`

职责：存放随应用交付的内置插件。

适合放置：

- 平台基础插件
- 默认示例插件
- 兼容性保底插件

# 5. 推荐的落地文件组织

## 5.1 目录组织方式

建议每个复杂子域都采用“目录 + 单一职责文件”方式，而不是把所有实现堆到一个大文件里。

例如：

```text
manifest/
  manifest.go
  types.go
  parser.go
  validator.go

discovery/
  scanner.go
  resolver.go
  planner.go

lifecycle/
  state.go
  manager.go
  transitions.go
  rollback.go
```

## 5.2 当前代码与目标目录的映射

### 现有目录

- `interface/` → 保留
- `loader/` → 保留并收敛职责
- `registry/` → 保留并扩展查询能力
- `application/` → 保留
- `domain/` → 保留
- `infrastructure/` → 保留
- `transport/` → 保留
- `builtin/` → 保留

### 新增目录

- `manifest/` → 新增
- `discovery/` → 新增
- `lifecycle/` → 新增
- `marketplace/` → 新增
- `runtime/` → 新增

# 6. 推荐的演进顺序

## Phase 1：契约稳定

先固化以下内容：

- 插件接口
- 注册器接口
- 清单结构
- 资源模型

## Phase 2：清单与发现

补齐：

- manifest 解析
- 插件扫描
- 兼容性校验
- 加载计划生成

## Phase 3：生命周期编排

补齐：

- 状态机
- 启用 / 禁用
- 加载 / 停止
- 升级 / 回滚

## Phase 4：市场化接入

补齐：

- 插件市场客户端
- 包下载与缓存
- 安装 / 卸载 / 升级
- 签名与校验

## Phase 5：运行时治理

补齐：

- 健康检查
- 资源挂载
- 热更新切换
- 审计与监控事件

# 7. 和现有实现的对齐建议

当前仓库已经存在插件元数据 CRUD 和 runtime registry，因此建议先做“目录补齐”，再做“大规模重构”。

## 7.1 先补能力，不先改边界

优先在现有目录中新增文件，逐步引入：

- manifest 解析
- discovery 扫描
- lifecycle 状态机
- marketplace 安装能力
- runtime 装配层

## 7.2 再做职责收拢

当新能力稳定后，再逐步把：

- 清单逻辑从 service 中拆出
- 资源装配从 loader 中拆出
- 安装升级从 transport 中拆出
- 持久化适配从 application 中拆出

---

# 8. 目录草案总结

如果只保留一句话，这份目录草案的核心就是：

> **manifest 管定义，discovery 管发现，lifecycle 管状态，marketplace 管分发，runtime 管运行，loader 管装载，registry 管注册。**

这套拆分方式更适合直接落地开发，也更适合后续做插件市场和热插拔能力。
