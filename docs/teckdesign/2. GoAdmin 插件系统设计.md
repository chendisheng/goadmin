# GoAdmin 插件系统设计 v1.0（Production Ready）

> **定位**：GoAdmin 平台级插件系统的独立设计文档。
>
> **目标**：把“插件能注册”升级为“插件可发现、可安装、可启停、可热切换、可治理、可分发”。
>
> **约束**：严格遵循 `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`，保持 clean-room，不复用 Gin-Vue-Admin 的源码、目录、接口、DTO、数据库结构或页面实现。

---

# 1. 设计目标

插件系统的核心目标不是简单扩展几个路由，而是为 GoAdmin 提供一套可持续演进的插件平台能力，使系统能够支持：

- 动态发现插件
- 插件安装 / 升级 / 卸载
- 插件启用 / 禁用
- 插件热切换 / 热更新
- 插件生命周期治理
- 插件菜单、路由、权限、事件的统一注册
- 插件市场化分发
- 插件运行态与插件包的解耦

插件系统必须遵守以下原则：

- **核心稳定，插件扩展**
- **先发现，再安装，再启用，再加载，再注册，再运行**
- **插件包是离线资产，运行态是在线资源**
- **插件失败不拖垮主系统**
- **插件能力必须可审计、可回滚、可治理**

# 3. 总体架构

## 3.1 运行链路

```text
插件来源
  ├── 内置插件 builtin/
  ├── 本地目录 local/
  ├── 已安装插件 installed/
  └── 插件市场 marketplace/
       ↓
Discovery（发现）
       ↓
Manifest（清单解析）
       ↓
Validation（依赖 / 版本 / 兼容性校验）
       ↓
Install（安装 / 落库 / 缓存）
       ↓
Lifecycle（启用 / 加载 / 初始化 / 注册 / 启动）
       ↓
Registry（路由 / 菜单 / 权限 / 事件 / 任务收敛）
       ↓
Runtime（运行态对外提供能力）
```

## 3.2 设计分层

插件系统建议拆成五个子域：

- `manifest`：描述插件是什么
- `discovery`：发现插件在哪里
- `lifecycle`：定义插件如何启停与切换
- `marketplace`：定义插件如何分发、安装与升级
- `runtime`：定义插件如何进入运行态

当前仓库已有的基础能力包括：

- 启动期显式加载插件
- 注册中心收敛路由、菜单、权限
- 插件元数据持久化与 CRUD 管理
- 启动时从注册中心种子同步到数据库

# 4. 目录结构建议

当前代码已经具备以下目录：

```text
backend/plugin/
  interface/
  loader/
  registry/
  application/
  domain/
  infrastructure/
  transport/
  builtin/
```

若要支撑动态发现、热插拔、生命周期治理和插件市场化，建议进一步演进为：

```text
backend/plugin/
  manifest/        # 清单结构、schema、解析器
  discovery/       # 扫描、索引、依赖解析
  lifecycle/       # 状态机、启停、切换编排
  marketplace/     # 市场目录、安装、升级、卸载
  runtime/         # 运行时装配与资源挂载
  loader/          # 插件装载
  registry/        # 运行时注册中心
  builtin/         # 内置插件
  application/
  domain/
  infrastructure/
  transport/
```

## 4.1 各目录职责

- `manifest/`：定义插件清单 schema、版本、依赖、能力边界
- `discovery/`：扫描插件来源并生成候选列表
- `lifecycle/`：统一管理安装、启用、加载、注册、启动、停止、卸载
- `marketplace/`：管理插件市场包、签名、下载、安装、升级、回滚
- `runtime/`：将插件资源挂载到应用运行态
- `loader/`：按顺序装载插件实例
- `registry/`：收敛插件对外注册的路由、菜单、权限和事件订阅
- `builtin/`：随应用发布的内置插件

# 5. 核心概念

## 5.1 插件包与插件运行态

必须明确区分两类对象：

- **插件包（Package）**：静态分发单元，包含代码、清单、签名、依赖、版本信息
- **插件运行态（Runtime Plugin）**：已经加载到当前进程中的插件实例

这一区分是支持市场化与热更新的前提。

## 5.2 插件清单

插件清单是插件进入系统的唯一可信描述，必须先于加载存在。

建议至少包含：

- `package_id`
- `name`
- `version`
- `description`
- `type`
- `source`
- `entrypoint`
- `enabled`
- `dependencies`
- `capabilities`
- `routes`
- `menus`
- `permissions`
- `events`
- `checksum`
- `signature`
- `compatible_app_version`

## 5.3 插件分级

```text
Core Plugin    → 平台级能力，如权限、菜单、配置、审计、身份集成
Business Plugin → 可独立演进的业务能力，如工单、审批、报表
External Plugin → 第三方集成能力，如短信、IM、支付、对象存储
```

# 7. 插件发现设计

## 7.1 发现来源

建议支持四种来源：

- `builtin`：内置插件，随应用发布
- `local`：本地插件目录，适合开发与灰度
- `installed`：数据库中已安装插件的启用状态
- `marketplace`：插件市场目录或远程包仓库

## 7.2 发现流程

1. 扫描候选插件
2. 读取并解析清单
3. 校验清单 schema
4. 解析依赖图
5. 校验版本与应用兼容性
6. 生成加载计划
7. 交给运行时注册中心执行

## 7.3 发现阶段应解决的问题

- 插件是否存在
- 插件是否完整
- 插件是否签名合法
- 插件是否与当前应用版本兼容
- 插件依赖是否满足
- 插件是否允许进入加载计划

# 8. 生命周期与热插拔

## 8.1 生命周期状态机

```text
Discovered
  → Installed
  → Enabled
  → Loaded
  → Initialized
  → Registered
  → Running
  → Stopping
  → Stopped
  → Disabled
  → Failed
```

## 8.2 生命周期动作

- `Discover`：发现并读取插件清单
- `Install`：落库、缓存包、写入元数据
- `Enable`：允许进入加载计划
- `Load`：装载代码或描述，建立运行上下文
- `Init`：注入配置、日志、容器、依赖
- `Register`：注册路由、菜单、权限、事件、定时任务
- `Start`：启动后台任务与订阅
- `Stop`：释放资源并优雅退出
- `Disable`：停止新流量并移出加载计划
- `Uninstall`：删除元数据与缓存包

## 8.3 热插拔策略

热插拔要分级处理：

- **支持热切换**：无状态路由、菜单、权限、事件订阅变更
- **谨慎热切换**：涉及后台任务、缓存、外部连接池的插件
- **禁止直接热切换**：涉及数据库结构迁移、核心鉴权链路、运行时全局状态的插件

## 8.4 热切换推荐流程

1. 标记目标插件为待切换
2. 停止对外接流或进入 drain 状态
3. 释放旧实例资源
4. 加载新版本插件
5. 重新注册运行时资源
6. 恢复对外服务
7. 若任一步失败，自动回滚到上一稳定版本

# 9. 失败、隔离与回滚

## 9.1 失败策略

- 核心平台插件失败：拒绝启动
- 业务插件失败：记录错误并跳过，不影响主程序
- 外部集成插件失败：允许降级，但必须告警
- 热更新失败：自动回滚到上一版本

## 9.2 隔离边界

- 路由前缀必须命名空间化，避免冲突
- 菜单、权限、事件名称必须带插件前缀
- 插件内部数据表与迁移脚本必须独立
- 插件资源配置应限制超时、并发与连接数
- 插件不得绕过主系统事务与审计约束

# 10. 插件市场化与分发

## 10.1 市场能力

插件市场建议支持：

- 插件列表检索
- 版本查询
- 安装 / 升级 / 卸载 / 启用 / 禁用
- 版本锁定与回滚
- 包签名与校验
- 兼容性检查
- 审核状态与发布状态

## 10.2 市场元数据

建议至少包含：

- `package_id`
- `name`
- `version`
- `publisher`
- `description`
- `category`
- `status`
- `download_url`
- `checksum`
- `signature`
- `compatible_app_version`
- `dependencies`
- `changelog`

## 10.3 市场交付路径

1. 市场检索
2. 下载包并校验
3. 安装到本地插件缓存区
4. 解析清单并做兼容性检查
5. 注册到生命周期管理器
6. 启用并发布到运行态

# 11. 运行时治理

## 11.1 资源治理

插件运行时资源必须可治理：

- 路由可收敛
- 菜单可刷新
- 权限可同步
- 事件可订阅 / 取消订阅
- 定时任务可启停

## 11.2 审计与可观测

插件状态变更应发布生命周期事件，供审计、市场、监控和自动化订阅：

- `PluginDiscovered`
- `PluginInstalled`
- `PluginEnabled`
- `PluginDisabled`
- `PluginLoaded`
- `PluginRegistered`
- `PluginStarted`
- `PluginStopped`
- `PluginUpgraded`
- `PluginUninstalled`
- `PluginHotReloadRequested`

# 12. 与现有 GoAdmin 代码的对齐

当前仓库已经具备的基础能力包括：

- 启动期显式加载插件
- 注册中心收敛路由、菜单、权限
- 插件元数据持久化与 CRUD 管理
- 启动时从注册中心种子同步到数据库
- 插件示例可注册路由、菜单与权限

现有实现已经说明：

- 插件不是仅仅“可调用”，而是已经具备**可装配与可持久化**基础
- 但还没有真正实现**自动发现、热更新、包级分发、版本治理**

因此，这份独立设计文档把下一阶段的实现目标明确为：

- `manifest/`：定义插件是什么
- `discovery/`：定义插件在哪里
- `lifecycle/`：定义插件如何启停与切换
- `marketplace/`：定义插件如何安装与分发
- `runtime/`：定义插件如何进入运行态

# 13. 推荐实施顺序

## Phase A：插件契约稳定

- 锁定 `Plugin` / `Registrar` / `Context` / `Menu` / `Permission` 模型
- 统一清单 schema
- 统一错误语义

## Phase B：发现与加载

- 实现本地目录扫描
- 实现 manifest 解析
- 实现依赖与版本校验
- 实现加载计划

## Phase C：生命周期管理

- 实现状态机
- 实现启用 / 禁用 / 升级 / 回滚
- 实现热切换流程

## Phase D：市场化能力

- 实现插件包索引
- 实现安装 / 卸载 / 升级接口
- 实现签名校验与兼容性检查

## Phase E：治理与可观测

- 实现生命周期事件
- 实现审计日志
- 实现插件指标与健康检查

---

# 14. 风险与约束

- 不允许插件直接修改核心鉴权链路
- 不允许插件绕过事务边界
- 不允许插件直接依赖内部实现细节
- 不允许未签名或未校验的市场包直接进入运行态
- 不允许热更新破坏主系统稳定性

---

# 15. 结论

GoAdmin 插件系统的目标不是做一个“能挂插件”的模块，而是做一套真正的平台级扩展基础设施：

- 能发现
- 能安装
- 能启停
- 能热切换
- 能治理
- 能分发
- 能回滚

这套设计可以直接指导后续 `manifest`、`discovery`、`lifecycle`、`marketplace`、`runtime` 的开发实现。
