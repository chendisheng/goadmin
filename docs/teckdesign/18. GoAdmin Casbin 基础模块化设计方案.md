# GoAdmin 授权治理模块基础模块化设计方案

> **定位**：将 GoAdmin 当前分散在 `core/auth/casbin`、`modules/casbin_model`、`modules/casbin_rule` 中的授权能力，提升为一个可装配、可治理、可迁移的基础模块。当前实现由 Casbin 提供，但模块边界必须保持可替换，以便未来切换授权运行时而不重写模块外壳。
>
> **依据**：本文档以 `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md` 为唯一架构来源，并承接 `docs/teckdesign/15. GoAdmin 基础模块变更权限检查清单.md` 的执行约束。
>
> **目标**：让当前由 Casbin 实现的授权能力，从“认证链路里的内部实现”升级为“平台级授权治理模块”，具备独立的模块边界、启动装配、策略治理、默认种子、管理入口与回归标准，同时保留未来切换授权运行时的抽象边界。

---

# 1. 背景与问题定义

## 1.1 当前现状

当前仓库中的授权能力主要分布在三处：

- `backend/core/auth/casbin`
  - 承担授权运行时、策略适配、默认规则、DB/file 双来源切换等核心逻辑（当前实现为 Casbin）。
- `backend/modules/casbin_model`
  - 提供授权模型的 CRUD 管理模块。
- `backend/modules/casbin_rule`
  - 提供授权策略规则的 CRUD 管理模块。

此外，`backend/modules/auth` 目前只负责登录、退出和 `me` 等认证入口，`core/auth/bootstrap` 负责把 JWT 和可替换授权运行时装配到运行时。

## 1.2 现有问题

这种形态可以运行，但还不算“真正的基础模块化”。主要问题是：

- **授权内核与管理模块割裂**
  - `core/auth/casbin` 负责运行时判断（当前实现为 Casbin）。
  - `casbin_model` / `casbin_rule` 负责数据管理。
  - 两者没有形成统一模块契约。
- **模块边界不完整**
  - 当前 Casbin 能力没有独立的模块清单、生命周期和依赖声明。
- **默认种子与运行时耦合过深**
  - 默认规则仍然硬编码在 enforcer 初始化逻辑中。
- **DB/file 双模式的治理能力不足**
  - 文件是源头还是数据库是源头，缺少清晰的“单一真相”语义。
- **缺少独立的权限治理入口**
  - 没有一个统一的授权治理模块负责策略查看、同步、重载、导入导出和审计。

## 1.3 升级目标

授权运行时升级为基础模块后，应满足：

- 既能作为平台级授权内核独立工作，又能被 `auth` 模块和其他基础模块消费。
- 既能通过 `file` 模式运行，又能通过 `db` 模式运行。
- 既能作为启动期基础能力装配，又能作为管理期可治理模块。
- 既能承载默认策略种子，又能对变更后的路由、菜单、权限资产做联动约束。

# 2. 设计原则

## 2.1 分层原则

Casbin 基础模块必须遵循 `transport -> application -> domain -> infrastructure` 的分层要求：

- `transport` 负责 HTTP / CLI / UI 入口。
- `application` 负责用例编排、同步流程和事务边界。
- `domain` 负责权限规则、模型、策略等核心概念。
- `infrastructure` 负责文件、数据库、watcher 等落地实现。

## 2.2 职责隔离原则

- `jwt` 只负责认证，不负责授权。
- `casbin` 只负责授权，不负责登录。
- `auth` 模块保留认证入口和运行时装配入口，不承载授权内核实现。
- `casbin_model` / `casbin_rule` 的数据管理能力，最终要收敛到统一的 Casbin 基础模块契约中。

## 2.3 可配置与可迁移原则

- 支持 `file` / `db` 双模式。
- 支持首次启动自动 seed。
- 支持策略同步与回填。
- 支持旧调用路径平滑迁移，不强制一次性切断。

## 2.4 一致性原则

- 路由、菜单、policy、默认种子、页面映射必须一致。
- 新增、修改、删除接口时，必须同步处理权限资产。
- 管理端看到的策略状态必须与运行时一致。

# 3. 目标架构

## 3.1 逻辑定位

授权运行时升级后的推荐定位是：

- **运行时内核**：`backend/core/auth/casbin`（当前实现为 Casbin）
- **模块级外壳**：`backend/modules/casbin`（建议的基础模块名，语义上为授权治理模块）
- **旧模块兼容层**：`backend/modules/casbin_model`、`backend/modules/casbin_rule`

其中：

- `core/auth/casbin` 保留引擎、适配器、加载器、重载器等底层实现。
- `modules/casbin` 负责把授权能力暴露为平台基础模块。
- `casbin_model` / `casbin_rule` 在迁移期可作为兼容入口，最终应被统一模块吸收。

## 3.2 推荐目录结构

```text
backend/
  core/
    auth/
      casbin/
        model/
          rbac.conf
          abac.conf
        enforcer/

    auth/

        repository/

        http/
          handler/
          request/
          response/
      module.go
```

## 3.3 模块边界定义

### 3.3.1 `core/auth/casbin`

负责：

- 加载模型文件。
- 初始化 enforcer。
- 维护规则缓存。
- 执行权限判断。
- 支持策略增删和重载。
- 支持 `file` / `db` 两种策略来源。

不负责：

- 页面管理。
- 菜单注册。
- 菜单权限渲染。
- 模块级路由清单组织。

### 3.3.2 `modules/casbin`

负责：

- 把 Casbin 作为平台基础模块对外注册。
- 提供策略、模型、同步、重载、导入导出等管理接口。
- 负责模块清单、菜单、权限和前端入口声明。
- 统一承接现有 `casbin_model` / `casbin_rule` 的管理能力。

### 3.3.3 `modules/auth`

负责：

- 登录、退出、当前用户信息。
- 认证态相关入口。
- 启动时调用 `core/auth/bootstrap` 完成 JWT + Casbin 装配。

不负责：

- 授权规则的业务治理。
- Casbin 管理页面。
- Casbin 策略 CRUD。

# 4. 核心能力设计

## 4.1 授权引擎

授权引擎继续以现有 `core/auth/casbin/enforcer` 为内核，但需要补齐三个能力：

- **模型校验**
  - 启动时检查模型文件完整性。
- **规则缓存**
  - 规则变更后可重载。
- **默认种子**
  - 初始环境自动注入最小可用策略。

引擎原则上不直接承载 HTTP 管理逻辑，只负责“可执行授权判定”。

## 4.2 策略存储

Casbin 基础模块应支持双来源：

- **file 模式**
  - `policy.csv` 作为主要策略来源。
  - 适合本地开发、默认 seed 和快速恢复。
- **db 模式**
  - 通过 Gorm 持久化到数据库。
  - 启动时自动迁移 `casbin_model`、`casbin_rule`。
  - 首次启动从文件 seed。

建议约定：

- 文件模式下，`policy.csv` 是直接源头。
- 数据库模式下，数据库是运行态源头，但 seed/回填仍要与文件保持可追溯一致。

## 4.3 默认策略种子

默认策略不应只是散落在 `enforcer.go` 中的硬编码数组，而应升级为模块资产。

建议默认种子至少覆盖：

- `auth/me`
- `auth/logout`
- 管理后台基础入口
- Casbin 自身的管理接口
- 关键基础模块的默认读权限

默认策略必须满足：

- 新环境可启动。
- 管理员可访问管理入口。
- 非授权角色不会被误放行。
- `file` / `db` 两种模式行为一致。

## 4.4 策略同步与重载

基础模块必须支持：

- 策略新增。
- 策略删除。
- 批量同步。
- 运行时重载。
- 变更后通知其他实例刷新。

在多实例场景下，策略更新应通过 watcher 或事件机制传播，避免单节点修改后状态不一致。

## 4.5 权限对象模型

推荐统一对象模型：

- `sub`：主体
- `obj`：资源
- `act`：动作
- `env`：上下文

推荐动作集合：

- `create`
- `read`
- `update`
- `delete`
- `export`
- `approve`

tenant 启用时，`sub` 可包含租户维度；tenant 关闭时退化为 role-only 语义。

# 5. 模块接口与对外契约

## 5.1 模块清单

Casbin 基础模块建议具备独立 `manifest.yaml`，至少声明：

- 模块名。
- 版本。
- 依赖模块。
- 路由。
- 菜单。
- 权限。
- 能力标签。

## 5.2 API 契约

建议至少提供以下能力：

- `GET /api/v1/casbin/models`
- `GET /api/v1/casbin/models/:id`
- `POST /api/v1/casbin/models`
- `PUT /api/v1/casbin/models/:id`
- `DELETE /api/v1/casbin/models/:id`
- `GET /api/v1/casbin/rules`
- `POST /api/v1/casbin/rules`
- `PUT /api/v1/casbin/rules/:id`
- `DELETE /api/v1/casbin/rules/:id`
- `POST /api/v1/casbin/reload`
- `POST /api/v1/casbin/seed`
- `GET /api/v1/casbin/status`

其中：

- `status` 用于查看当前模式、模型、规则同步状态。
- `reload` 用于重载运行时规则。
- `seed` 用于补齐默认策略。

## 5.3 菜单与前端入口

Casbin 基础模块应在系统菜单中拥有独立入口，例如：

- 授权管理
- 模型管理
- 策略管理
- 同步状态

前端页面建议与后端模块一致，避免 `menu permission`、API 路径、页面名称分离成三套命名体系。

## 5.4 权限资产

Casbin 自身的管理接口也必须受 Casbin 管控。

也就是说，Casbin 基础模块需要先解决“自己怎么被管理”的问题：

- 管理员默认可见。
- 非管理员按策略授权。
- 新增接口必须补策略。
- 删除接口必须移除失效策略。

# 6. 启动装配与生命周期

## 6.1 启动流程

推荐启动顺序：

1. 加载配置。
2. 初始化数据库。
3. 初始化 JWT。
4. 初始化 Casbin 内核。
5. seed 默认策略。
6. 注册 Casbin 基础模块。
7. 注册其他业务模块。
8. 挂载路由与菜单。

## 6.2 装配方式

`core/auth/bootstrap` 负责把 JWT 和 Casbin 运行时对象装配出来，但不应承担模块管理职责。

模块装配层应负责：

- 调用 Casbin 模块的初始化。
- 注册管理路由。
- 注册默认菜单。
- 注册权限资源。
- 订阅策略变更事件。

## 6.3 生命周期

Casbin 基础模块至少应具备以下生命周期语义：

- `Init`
- `Register`
- `Seed`
- `Reload`
- `Sync`
- `Stop`

这能保证它不只是一个静态库，而是一个可以被平台治理的模块。

# 7. 数据模型与迁移策略

## 7.1 数据归属

建议保留以下资产：

- `casbin_model`
  - 存储模型内容与版本信息。
- `casbin_rule`
  - 存储策略规则。
- `policy.csv`
  - 作为文件种子或 file 模式源头。

## 7.2 迁移原则

- 模型表与策略表迁移必须幂等。
- 默认种子必须幂等。
- 规则同步必须允许重复执行。
- 删除失效规则时必须明确“是否删除历史 DB 记录”。

## 7.3 兼容策略

迁移期建议保留：

- 旧的 `casbin_model` / `casbin_rule` CRUD 模块。
- 新的统一 `casbin` 基础模块。

两者可通过兼容路由或内部复用共存一段时间，待调用方迁移完成后再逐步收敛。

---

# 8. 与其他基础模块的协作

## 8.1 与 `auth` 模块

- `auth` 负责认证入口。
- `casbin` 负责授权治理。
- `auth` 在登录后把 claims 交给 Casbin 做权限判断。

## 8.2 与 `user` / `role` / `menu`

- 用户、角色、菜单变更时，必须同步检查权限资产。
- 菜单权限与 API 权限必须一致。
- 角色变更后需要重评估策略覆盖范围。

## 8.3 与 `plugin`

- 插件若新增路由、菜单或权限资源，必须注册到 Casbin。
- 插件卸载时要清理对应策略。
- 插件与 Casbin 的耦合点应通过模块契约，而不是直接修改内核实现。

## 8.4 与 `codegen`

- CodeGen 生成 CRUD 时，应同步生成 Casbin policy。
- 生成模块安装时，策略、菜单、路由要一起落地。
- 删除生成模块时，要同步清理相关权限资源。

---

# 9. 迁移实施路径

## 9.1 第一阶段：内核收敛

- 保持 `core/auth/casbin` 不变。
- 抽清职责边界。
- 统一 `file/db` 行为。
- 把默认规则整理为模块化资产。

## 9.2 第二阶段：模块化承载

- 新建 Casbin 基础模块清单。
- 将模型、策略、同步、重载纳入统一服务。
- 让管理 API / 菜单 / 页面全部挂到模块契约上。

## 9.3 第三阶段：兼容迁移

- `casbin_model` / `casbin_rule` 改为兼容入口或内部子模块。
- 逐步迁移调用方到统一模块。
- 收敛重复 API 和重复菜单。

## 9.4 第四阶段：治理闭环

- 增加策略变更审计。
- 增加同步状态检查。
- 增加升级/回滚方案。
- 增加 delete / reload / seed 的回归测试。

---

# 10. 风险与约束

## 10.1 风险

- **双来源同步风险**
  - file 与 db 并存时容易出现不一致。
- **默认策略漂移风险**
  - 代码、文件、数据库三份资产可能不同步。
- **管理接口死锁风险**
  - Casbin 自己的管理接口若缺少默认授权，会导致无法管理。
- **历史策略残留风险**
  - 删除接口后，如果历史策略没清理，会继续放行旧路径。

## 10.2 约束

- 不允许绕过 `auth.casbin.source` 的配置语义。
- 不允许把权限判断写回 handler 中。
- 不允许让 `casbin` 直接依赖别的模块的 infrastructure 实现。
- 不允许只改代码不改 policy、菜单和默认种子。

---

# 11. 验收标准

Casbin 被视为“真正升级为基础模块”，至少要满足以下条件：

- **可装配**
  - 启动时能作为模块被注册和初始化。
- **可治理**
  - 有统一的策略管理 API / 页面 / 菜单。
- **可迁移**
  - `file` / `db` 双模式可切换，且行为一致。
- **可同步**
  - 默认种子、DB 数据、文件策略可保持一致。
- **可回归**
  - 新增、删除、修改接口时，权限资产联动检查有测试覆盖。
- **可审计**
  - 策略变更和模块变更可追踪。

---

# 12. 结论

Casbin 的“真正升级为基础模块”，不是简单把 `policy.csv` 和 `casbin_rule` 再包一层，而是把**授权内核、策略存储、默认种子、启动装配、管理入口、变更同步、回归约束**一起收敛成一个完整的平台级模块。

在 GoAdmin 的架构里，推荐保留 `core/auth/casbin` 作为底层内核，同时新增统一的 Casbin 基础模块承载对外治理能力，并逐步收敛 `casbin_model` / `casbin_rule` 这两个分散的管理入口。这样才算真正完成从“内部实现”到“基础模块”的升级。
