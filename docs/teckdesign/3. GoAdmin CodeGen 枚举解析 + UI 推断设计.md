# GoAdmin CodeGen 枚举解析 + UI 推断设计 v1.0（Production Ready）

> **定位**：GoAdmin CodeGen 的枚举识别、归一化与前端 UI 推断专项设计文档。
>
> **目标**：让 CodeGen 能够从 **DSL** 与 **数据库表** 中稳定提取模型字段枚举信息，并在生成前端表单、筛选器、详情与列表展示时自动推断合适的 UI 组件。
>
> **约束**：
>
> - 严格遵循 `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`
> - 保持 clean-room，不复用 Gin-Vue-Admin 的源码、模板、接口、表结构或生成规则
> - 枚举能力必须支持增量演进，不得破坏现有 `backend/codegen` 生成链路
> - DSL / DB 两条输入链路必须共享同一套归一化规则
> - UI 推断必须是确定性的、可覆盖的、可预览的

---

# 1. 背景与问题

当前 CodeGen 已经具备字段级基础映射能力：

- `backend/codegen/schema/database/model.go` 已预留 `Column.EnumValues`
- `backend/codegen/model/ir/ir.go` 已预留 `Field.EnumValues` 与 `Semantic.EnumFields`
- `backend/codegen/application/irbuilder/service.go` 会在存在枚举值时把 UI 类型推断为 `select`
- `backend/codegen/application/irbuilder/semantic.go` 会把枚举字段归入 `enum` 语义

但这些能力仍然只覆盖“最小可用”的场景，距离生产级还有明显差距：

- DSL 侧还没有正式的枚举表达方式
- DB 侧还没有统一的枚举提取协议
- 只有 `select` 一种粗粒度 UI 推断，缺少 `radio`、`switch`、`checkbox-group`、`tag`、`autocomplete` 等场景化映射
- 缺少枚举来源、置信度、冲突和回退的统一规则
- 缺少对“字典型枚举”与“静态枚举”的区分
- 缺少对预览报告、生成报告和前端渲染的统一数据契约

因此，需要一份专门的专项设计文档来定义：

- 枚举从哪里来
- 怎么解析
- 怎么归一化
- 怎么合并冲突
- 怎么推断 UI
- 怎么让生成器消费
- 怎么保持兼容性与可扩展性

---

# 2. 设计目标

## 2.1 核心目标

- **统一枚举来源**：DSL、DB 原生类型、约束、字典引用都能进入统一模型
- **统一数据契约**：所有枚举信息最终落到 IR 的统一字段结构中
- **统一 UI 推断**：前端生成器不直接读数据库类型，而是消费标准化的枚举语义
- **统一冲突处理**：对重复值、空值、类型冲突、大小写冲突提供稳定规则
- **统一可观测性**：预览报告能够解释为什么推断成某种 UI 组件
- **统一兼容性**：保留当前 `EnumValues` / `Semantic.EnumFields` 作为最小兼容层

## 2.2 非目标

- 不在 CodeGen 中实现运行时字典服务本身
- 不把业务状态机逻辑硬编码进生成器
- 不强制所有枚举都必须走数据库反查
- 不要求所有前端框架共享相同的具体组件实现
- 不把 UI 推断直接绑定到某个页面库的私有 API

# 3. 当前实现现状

## 3.1 现有基础

当前仓库已经具备以下基础能力：

- `schema/database.Column` 有 `EnumValues []string`
- `model/ir.Field` 有 `EnumValues []string`
- `model/ir.Semantic` 有 `EnumFields map[string][]string`
- `irbuilder` 会在枚举存在时将 `SemanticType` 设为 `enum`
- `irbuilder` 会把 `UIType` 推断为 `select`
- DB preview 输出已经能携带 `EnumValues`

## 3.2 现有差距

- `schema/parser.go` 的 `rawField` 还没有枚举结构
- DB inspector 侧没有形成统一的 `EnumExtractor` 协议
- `UIType` 现在是单字符串，缺少页面上下文与控件策略信息
- 预览信息里还没有“来源说明”和“推断理由”
- 生成器缺少针对多选、标签、字典远程加载的差异化渲染策略

# 4. 设计原则

## 4.1 单一事实来源

枚举能力必须从统一事实来源进入 IR，而不是在模板层临时判断。

推荐顺序：

- DSL 显式定义
- DB 原生结构提取
- DB 约束提取
- 字典引用提取
- 规则覆盖
- 自动推断兜底

## 4.2 显式优先于推断

如果 DSL 或元数据中显式声明了枚举信息，必须优先于自动推断。

## 4.3 兼容优先

当前的 `EnumValues []string` 仍然是有效最小契约，不能因为引入更强表达而破坏已有链路。

## 4.4 UI 推断必须可解释

每个推断结果都应该能说明：

- 来源是什么
- 规则是什么
- 为什么选这个控件
- 为什么没有选其他控件

## 4.5 前端适配器化

CodeGen 只输出“UI 意图”，不把某个 UI 库的组件名写死在核心模型中。

# 5. 枚举统一模型

## 5.1 建议引入的抽象

为了让枚举在 DSL、DB 和 UI 三层都能稳定复用，建议把枚举抽象为三个层次：

- **EnumOption**：单个枚举项
- **EnumSpec**：一个字段的枚举定义
- **EnumUIHint**：基于枚举生成的 UI 推断结果

## 5.2 EnumOption

单个枚举项建议至少包含：

- `value`：存储值
- `label`：展示名
- `color`：视觉标识，可选
- `disabled`：是否禁用，可选
- `order`：排序权重，可选
- `metadata`：扩展信息

## 5.3 EnumSpec

字段级枚举定义建议至少包含：

- `kind`：`static` / `database_native` / `constraint` / `dictionary` / `override`
- `mode`：`single` / `multiple`
- `options`：枚举项列表
- `source_ref`：来源引用
- `display`：推荐展示模式
- `confidence`：推断置信度
- `fallback`：回退策略
- `metadata`：扩展信息

## 5.4 EnumUIHint

UI 推断结果建议至少包含：

- `input_control`：表单控件类型
- `table_renderer`：列表/详情展示方式
- `search_control`：搜索区控件类型
- `multiple`：是否多选
- `remote`：是否远程加载
- `filterable`：是否可搜索
- `clearable`：是否可清空
- `tag_style`：是否使用标签/徽标展示
- `source_reason`：推断原因说明

# 6. 输入源设计

## 6.1 DSL 枚举设计

### 6.1.1 目标语法

DSL 应支持显式声明枚举，而不是只靠字段类型猜测。

建议采用以下主结构：

```yaml
fields:
  - name: status
    type: string
    enum:
      kind: static
      mode: single
      display: select
      options:
        - value: draft
          label: 草稿
          color: gray
        - value: published
          label: 已发布
          color: green
```

### 6.1.2 兼容语法

为了兼容简单场景，允许保留轻量写法：

```yaml
fields:
  - name: status
    type: string
    enum_values:
      - draft
      - published
```

兼容语法的处理规则：

- 自动补齐 `label = value`
- 默认 `kind = static`
- 默认 `mode = single`
- 默认 `display = select`

### 6.1.3 字典型枚举

对于共享字典、系统字典或可复用状态码，DSL 可以显式引用字典源：

```yaml
fields:
  - name: category_code
    type: string
    enum:
      kind: dictionary
      source_ref: system.dictionary.category
      display: remote-select
```

这类枚举不把所有选项固化进代码，而是通过字典源在生成时或运行时解析。

## 6.2 DB 枚举设计

DB 侧枚举提取必须支持“从数据库结构中反向识别枚举语义”。

### 6.2.1 数据库原生枚举

适用于：

- MySQL `ENUM`
- PostgreSQL `ENUM`
- 其他具有原生枚举类型的方言

规则：

- 直接提取枚举值列表
- 保留原始顺序
- 值与标签默认相同
- 方言信息写入 metadata，便于调试和报告输出

### 6.2.2 约束型枚举

适用于：

- `CHECK (col IN (...))`
- `CHECK (col = ANY(...))`
- 其他可解析的枚举约束

规则：

- 仅在约束可被明确解析时提取
- 若解析失败，不应误报为枚举
- 必须记录推断来源为 `constraint`

### 6.2.3 字典引用型枚举

适用于：

- 外键指向字典表
- 列注释或元数据中声明字典编码
- 业务字段映射到系统字典

规则：

- 若能解析到字典表或字典编码，优先视为 `dictionary`
- `value` 为字典编码或字典项值
- `label` 为字典项名称
- 若字典项可远程加载，UI 优先使用 `remote-select`

### 6.2.4 元数据型枚举

适用于：

- 数据库导出 DSL
- 数据库侧配置表
- 迁移后的结构化元数据

规则：

- `Column.Metadata` 可携带 `enum_values`、`enum_options`、`enum_kind` 等扩展信息
- `Column.EnumValues` 保持最小兼容层
- 生成链路不得丢失已识别的枚举信息

### 6.2.5 注释解析型枚举

适用于：

- 非原生枚举类型，但列注释中显式描述了枚举取值
- 例如 `status tinyint comment '0=禁用,1=启用'`
- 例如 `gender int comment '1=男,2=女'`
- 例如 `type varchar comment 'book,video,audio'`

规则：

- 注释只在符合机器可解析格式时才参与枚举推断，避免把普通说明文字误判为枚举
- `value=label` 形式优先，`value` 与 `label` 都会进入枚举项
- 若仅给出值列表，则 `label` 默认等于 `value`，必要时在前端按人类可读规则做轻量美化
- 也支持“说明前缀 + 枚举串”的写法，例如 `分类|tech=技术,novel=小说,history=历史,other=其他`，其中左侧说明仅用于人读，右侧作为可解析枚举串
- 注释解析得到的枚举优先级低于 DSL 显式枚举和 DB 原生枚举，高于纯启发式推断
- 解析结果必须写入 metadata，记录来源为 `comment`

推荐识别形式：

- `0=禁用,1=启用`
- `1=男,2=女`
- `book,video,audio`
- `分类|tech=技术,novel=小说,history=历史,other=其他`

不推荐的形式：

- 混合自然语言说明与枚举值，例如 `状态：0=禁用，1=启用，默认 0`
- 同时包含多段不同语义的描述，例如 `A/B 测试状态 0=关,1=开，历史值见字典表`
- 使用不稳定分隔符、容易歧义的格式，例如 `0|禁用;1|启用`

### 6.2.6 注释约定建议

为了让注释既适合人读，也适合生成器稳定解析，建议统一采用以下约定：

- **建议加显式前缀**：优先写成 `enum: 0=禁用,1=启用`
- **支持描述前缀**：也可以写成 `分类|tech=技术,novel=小说,history=历史,other=其他`，左侧描述仅用于人读，右侧作为可解析枚举串
- **一行一义**：注释只描述这一列的枚举，不混入业务说明、默认值说明或迁移备注
- **优先使用 value=label**：当展示文案与存储值不同的时候，优先显式给 label
- **纯值列表仅作简化写法**：只有在 label 与 value 相同或可直接推断时才使用 `book,video,audio`
- **避免自由文本**：除非显式前缀存在，否则不要写成带自然语言的长句
- **保持分隔符统一**：推荐逗号分隔枚举项，等号分隔 value 和 label

推荐示例：

```sql
status tinyint comment 'enum: 0=禁用,1=启用'
gender int comment 'enum: 1=男,2=女'
type varchar(32) comment 'enum: book,video,audio'
category varchar(255) not null comment '分类|tech=技术,novel=小说,history=历史,other=其他'
```

如果需要补充说明，建议把说明放到单独字段注释或 DSL metadata 中，不要写进枚举串本体。

# 7. 归一化与合并规则

## 7.1 归一化步骤

枚举从输入到 IR 的建议流程：

1. **采集**：从 DSL 或 DB 收集原始枚举数据
2. **识别**：判断是否属于枚举候选
3. **归一化**：把不同来源统一成 `EnumSpec`
4. **校验**：检查值唯一性、类型一致性、空值等问题
5. **合并**：融合 DSL 覆盖、规则覆盖、字典覆盖
6. **降级**：若冲突不可接受，则回退到普通字段语义
7. **输出**：写入 IR、预览报告和前端生成上下文

## 7.2 合并优先级

建议优先级如下：

- **第一优先**：DSL 显式枚举定义
- **第二优先**：字段级显式覆盖规则
- **第三优先**：DB 原生枚举
- **第四优先**：DB 约束枚举
- **第五优先**：字典引用
- **第六优先**：启发式推断

## 7.3 去重与排序

- 相同 `value` 只保留第一次出现的定义
- 若同值但 label 不同，记录冲突警告并保留显式来源优先项
- 默认保留源顺序
- 若存在显式排序字段，按排序字段重排

## 7.4 数据校验

必须校验以下问题：

- 枚举值为空
- 枚举值重复
- 值类型与字段类型不兼容
- 多选枚举却被生成到不支持多选的控件
- 字典引用无法解析
- 约束解析结果与显式定义冲突

校验失败时不应直接终止整个生成流程，除非该字段是显式强约束字段；默认应降级并发出可读警告。

# 8. UI 推断规则

## 8.1 总体原则

UI 推断不是“把枚举都变成 select”这么简单，而是要根据：

- 枚举来源
- 项目语义
- 值数量
- 是否多选
- 页面场景
- 是否需要远程加载
- 是否适合标签展示

进行分层决策。

## 8.2 表单场景

### 8.2.1 单选静态枚举

推荐：

- 小型集合：`radio`
- 常规集合：`select`

适用条件：

- `mode = single`
- `options` 数量较少
- 无远程加载需求

### 8.2.2 多选枚举

推荐：

- `checkbox-group`
- 或 `multiple select`

适用条件：

- `mode = multiple`
- 字段存储支持数组、分隔串或关联表

### 8.2.3 二值枚举 / 布尔化枚举

推荐：

- `switch`
- 或 `radio`

适用条件：

- 仅两个稳定状态
- 语义接近开关、启用/禁用、是/否

### 8.2.4 字典远程枚举

推荐：

- `remote-select`
- 或 `autocomplete`

适用条件：

- 选项数量大
- 选项来源不适合静态写死
- 需要共享字典或跨模块复用

## 8.3 搜索场景

搜索区 UI 应优先兼顾效率：

- 小枚举：`select`
- 大枚举：`autocomplete` 或 `remote-select`
- 二值状态：`switch` 或 `select`
- 多选筛选：`multiple select`

## 8.4 列表展示场景

当字段被识别为枚举时，列表页默认不再直接输出原始裸值，而是优先输出可读标签或状态样式：

- `status` / `boolean-like`：优先 `tag` / `badge`
- 普通分类枚举：优先 `tag`，必要时回退为文本标签
- 值列表来自注释、DB 原生枚举或 DSL 显式枚举时，列表展示规则保持一致

对于你给出的这三类字段：

- `status tinyint comment '0=禁用,1=启用'`：识别为枚举，表单优先 `select` 或 `radio`，列表优先 `tag`
- `gender int comment '1=男,2=女'`：识别为枚举，表单优先 `select`，列表优先 `tag`
- `type varchar comment 'book,video,audio'`：识别为静态枚举，表单优先 `select`，列表优先 `tag` 或文本标签

这意味着它们“在 UI 上都能以列表形式展示”，但表单控件与列表展示控件可以不同：

- 表单：用于选择，强调输入效率
- 列表：用于浏览，强调可读性

## 8.4 列表场景

列表页推荐使用展示型 UI：

- `tag`
- `badge`
- `text`
- `status pill`

优先级建议：

- 状态型枚举：`tag` 或 `badge`
- 普通分类枚举：`text`
- 需要颜色语义时：`tag`

## 8.5 详情场景

详情页建议：

- 复用列表展示策略
- 保持值与标签清晰区分
- 对字典型枚举展示 `label`，必要时附带 `value`

## 8.6 UI 推断矩阵

- **enum + single + 小集合** → `radio` / `select`
- **enum + single + 大集合** → `select` / `autocomplete`
- **enum + multiple** → `checkbox-group` / `multiple select`
- **status** → `select` + `tag` 展示
- **dictionary** → `remote-select` + `tag` 展示
- **boolean-like** → `switch`
- **无枚举但命中状态规则** → 允许降级为 `select`

## 8.7 组件映射建议（Vue3 / Element Plus）

前端适配器应把 UI 意图映射为具体组件：

- `select` → `el-select`
- `radio` → `el-radio-group`
- `switch` → `el-switch`
- `checkbox-group` → `el-checkbox-group`
- `autocomplete` → `el-autocomplete`
- `remote-select` → `el-select` + remote loading
- `tag` → `el-tag`
- `badge` → `el-badge`

核心模型不直接依赖这些组件名，而是只输出 UI 意图。

# 9. 与现有 CodeGen 链路的协作

## 9.1 IR Builder

现有 `irbuilder` 应继续承担“字段语义归一化”的职责，但需要升级为：

- 先做枚举解析
- 再做语义推断
- 再做 UI 意图选择
- 再写入 Metadata

建议输出内容至少包括：

- `Field.EnumValues`
- `Field.SemanticType = enum` 或 `status`
- `Field.UIType`
- `Field.Metadata["enum_source"]`
- `Field.Metadata["enum_kind"]`
- `Field.Metadata["enum_confidence"]`
- `Field.Metadata["ui_reason"]`

## 9.2 Semantic 层

`Semantic` 层建议继续保留 `EnumFields`，但应增加面向报告和 UI 的补充信息：

- 枚举来源统计
- 状态字段优先级
- 字典引用数量
- 远程加载建议

## 9.3 前端生成器

前端生成器应基于枚举语义生成：

- 表单字段控件
- 搜索区控件
- 列表展示格式化器
- 详情页展示格式化器
- 初始值与清空逻辑
- 远程字典加载逻辑

## 9.4 预览与报告

DB 预览 / DSL 预览应暴露：

- 字段枚举来源
- 枚举项数量
- 推断出的 UI 类型
- 备选 UI 类型
- 冲突与降级说明

这样用户在真正生成前就能确认枚举是否被正确识别。

---

# 10. 冲突、降级与回退

## 10.1 冲突处理

当多个来源同时提供枚举定义时：

- 显式 DSL 定义覆盖所有自动推断
- 显式规则覆盖 DB 推断
- DB 原生枚举优先于约束解析
- 字典引用优先于简单静态值列表

## 10.2 降级策略

若出现以下问题，应自动降级：

- 枚举值不完整
- 字典不可解析
- 类型不兼容
- 枚举项过多且无法远程加载
- UI 组件不支持当前模式

降级后应回退为普通文本输入或基础 select，并在预览报告中给出原因。

## 10.3 回退优先级

建议回退顺序：

1. `remote-select`
2. `select`
3. `radio`
4. `text input`

这样可以尽量保证生成物可用，而不是因为枚举信息不完整导致整页不可生成。

# 11. 生产级要求

## 11.1 确定性

同一份 DSL 或同一张表，在相同规则下必须生成同样的枚举结果和 UI 推断结果。

## 11.2 可解释

预览报告必须说明：

- 枚举从哪来
- 为何被识别为枚举
- 为何选择该 UI 组件
- 为何没有采用其他控件

## 11.3 可扩展

后续新增的枚举来源不应修改已有核心语义，只需新增解析器或适配器。

## 11.4 可测试

必须能通过单测覆盖：

- DSL 显式枚举解析
- DB 原生枚举解析
- 约束枚举解析
- 字典引用解析
- UI 推断矩阵
- 冲突与降级

## 11.5 可观测

建议在预览与生成报告中统计：

- 枚举字段数量
- 静态枚举数量
- 字典枚举数量
- 远程加载建议数量
- 降级数量
- 冲突数量

# 12. 推荐实施分期

## Phase 1：模型与 DSL 承载

- 扩展 DSL 字段结构，支持 `enum` / `enum_values`
- 统一 `EnumOption` / `EnumSpec` 的内部表示
- 保留现有 `EnumValues` 兼容路径

## Phase 2：DB 枚举提取

- 实现数据库原生枚举提取
- 实现约束枚举提取
- 支持字典引用识别
- 统一写入 `Column.EnumValues` 和 `Column.Metadata`

### 注释约定落地

- 解析列注释时，仅把满足机器可解析格式的内容视为枚举候选，避免普通说明文字被误判
- 优先识别带显式前缀的写法，例如 `enum: 0=禁用,1=启用`
- 兼容 `value=label` 与纯值列表写法，例如 `1=男,2=女`、`book,video,audio`
- 其余注释内容一律按普通说明处理，不进入枚举推断结果

## Phase 3：IR 与语义层升级

- 统一字段枚举元数据
- 扩展 `Semantic` 的报告能力
- 输出枚举来源、置信度与 UI 推断原因

## Phase 4：前端 UI 推断

- 根据枚举场景生成表单控件
- 增加列表/详情标签渲染
- 增加远程字典加载支持
- 支持搜索区枚举控件推断

## Phase 5：预览与测试

- 将枚举来源与 UI 推断纳入 dry-run / preview 报告
- 增加回归测试与快照测试
- 验证 DSL/DB 双驱动结果一致性

# 13. 实施边界建议

建议按以下目录逐步落地：

- `backend/codegen/schema/`：DSL 枚举解析、DB 枚举提取协议
- `backend/codegen/application/irbuilder/`：枚举归一化与 UI 推断
- `backend/codegen/model/ir/`：IR 枚举结构扩展
- `backend/codegen/driver/cli/`：预览报告与执行链路透传
- `backend/codegen/generator/frontend/vue3/`：枚举控件渲染与字典加载

---

# 14. 结论

GoAdmin CodeGen 的枚举能力不能只停留在“有一个字符串列表”，而应该形成一条完整链路：

- **输入侧**支持 DSL、DB 原生结构、约束、字典引用
- **中间层**统一归一化为可解释的枚举模型
- **语义层**输出稳定的字段语义与 UI 意图
- **生成层**自动渲染合适的前端控件
- **报告层**清晰说明来源、推断和降级原因

只要按这份设计实现，CodeGen 就能把枚举处理从“简单 select 映射”升级为真正可用于生产的智能 UI 推断能力。
