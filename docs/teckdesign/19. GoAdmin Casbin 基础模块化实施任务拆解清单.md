# GoAdmin Casbin 基础模块化实施任务拆解清单

> **依据文档**：
> - `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`
> - `docs/teckdesign/15. GoAdmin 基础模块变更权限检查清单.md`
> - `docs/teckdesign/18. GoAdmin Casbin 基础模块化设计方案.md`
>
> **核心原则**：
> - Casbin 必须从“认证链路里的内部实现”升级为“可装配、可治理、可迁移的基础模块”
> - `core/auth/casbin` 继续作为基础模块的授权底座与数据库适配核心，仍是 Casbin 基础模块的一部分，不退化为纯内部工具
> - `modules/casbin` 作为基础模块的管理外壳承接对外能力；`casbin_model` / `casbin_rule` 在迁移期保留兼容入口，但不再作为新的命名基准
> - 数据库表命名保持统一风格，本文档中的正式表名统一采用单数风格：`casbin_model` / `casbin_rule`；`casbin_models` / `casbin_rules` 仅作为历史兼容或资源语义保留
> - `file` / `db` 两种策略来源必须保持行为一致，且 `policy.csv` 仍然是文件侧的源头
> - 新增、删除、修改权限资产时，必须同步检查路由、菜单、policy、默认种子与前端映射
> - 授权运行时必须通过可替换抽象接口暴露给模块层，避免业务代码直接依赖 Casbin 名称，确保未来切换鉴权引擎时只需替换实现而不重写模块外壳
> - 严格保持 clean-room，不复用 Gin-Vue-Admin 的源码、接口签名、目录结构与实现逻辑

---

# 1. 实施顺序建议

建议按照下面顺序推进：

1. 先定义 Casbin 基础模块的模块边界、清单和对外契约
2. 再收敛 `core/auth/casbin` 的授权内核职责，明确 file / db 双模式统一行为
3. 再落地新的 `modules/casbin` 模块外壳与应用层服务
4. 再接入管理 API、菜单、权限与前端页面
5. 再做默认策略种子、启动装配与同步机制
6. 再处理 `casbin_model` / `casbin_rule` 的兼容迁移与收敛
7. 最后补齐测试、文档与实施收尾

### 主要工作
- 梳理 `enforcer`、`adapter`、`service` 的职责边界
- 将模型校验、规则缓存、默认种子和重载逻辑整理为统一内核能力
- 明确 `file` / `db` 双模式的统一初始化路径
- 规范 `PermissionService` / `Authorizer` 对外接口
- 清理不属于授权内核的业务耦合
- 保证与 `core/auth/bootstrap` 的装配接口稳定

### 交付物

- 收敛后的 `core/auth/casbin/adapter`

### 完成标准
- `core/auth/casbin` 只保留授权引擎相关能力
- `file` / `db` 两种模式的初始化、seed 和 reload 行为一致
- 内核不再直接承担 UI、菜单或模块清单职责

- 复用 `core/auth/casbin` 作为内核，不重复实现鉴权引擎

- `server/modules/casbin/domain/`

- Casbin 可以通过模块级服务完成治理动作
- 应用层不直接依赖具体 UI 或具体框架细节

# 3. 实施完成标准

## 3.1 功能完成标准

- Casbin 能作为独立基础模块被注册和初始化
- 授权内核、模块管理、菜单入口和权限资产形成统一闭环
- `file` / `db` 两种策略模式可以一致运行
- 默认策略、同步、重载、状态查询能力齐备

## 3.2 迁移完成标准

- `casbin_model` / `casbin_rule` 可平滑迁移或收敛
- 旧调用路径仍可用，但不再作为主入口
- 新模块成为权限治理的唯一推荐入口
- 数据库正式表名只保留一套统一风格，兼容表或旧表名仅用于迁移期

## 3.3 工程完成标准

- 重复执行不产生重复菜单、重复 policy 或重复 seed
- 失败能定位到阶段
- 共享资源不会被误删
- 人工资产不会被默认覆盖

## 3.4 文档完成标准

- Casbin 模块边界清晰
- 迁移路径清晰
- 与 `15. GoAdmin 基础模块变更权限检查清单.md` 的联动关系清晰
- 团队可以按任务清单直接进入开发

# 4. 收尾建议

建议实际推进顺序如下：

1. 先做任务 1 和任务 2，明确边界并收敛内核
2. 再做任务 3，建立统一模块外壳
3. 再做任务 4 和任务 5，补齐治理与运行时装配
4. 再做任务 6，处理旧模块迁移收敛
5. 最后做任务 7，补齐测试与文档收尾

# 5. 结论

这份实施清单的核心目标，不是简单把 Casbin 再包一层，而是把它真正升级成 GoAdmin 的**平台级基础模块**，同时保留可运维的维护页和可治理的管理页。

如果按这份清单推进，最终会形成：

- `core/auth/casbin`：基础模块的授权底座与数据库适配核心
- `modules/casbin`：基础模块外壳与管理入口
- `casbin_model` / `casbin_rule`：迁移期兼容入口与正式单数命名基准
- 维护页与管理页：保留，但管理页去除 codegen 标识

这样才能同时满足架构一致性、模块化治理和后续扩展能力。
