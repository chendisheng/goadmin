# GoAdmin 架构设计 v1.1

> **定位**：面向企业级后台管理平台的 **clean-room** 架构设计文档。
>
> **对标目标**：只对标 Gin-Vue-Admin 的能力形态，不复用其源码、目录组织、接口签名、字段命名、页面实现或数据库设计。
>
> **核心方向**：**模块化单体 + 插件化扩展 + 事件驱动 + 多租户预留 + 多框架适配 + DevOps 交付**。

---

# 1. 文档说明

## 1.1 文档目标

本文档用于定义 GoAdmin v1.1 的总体架构、目录规范、扩展方式与分阶段实施路线，目标是让开发团队可以直接据此落地实现，而不是停留在概念层。

## 1.2 适用范围

本文档覆盖：

- 后端核心架构
- 模块边界与依赖规则
- 插件系统
- 事件总线
- 多租户设计
- 权限与鉴权
- gin / go-zero / kratos 适配方式
- 前端工程组织方式
- DevOps 与交付规范
- 分阶段实施计划

## 1.3 Clean-Room 声明

GoAdmin 只参考公开可见的工程范式与通用架构思想，不复用任何既有实现。

必须遵守：

- 不复制原项目源码
- 不沿用原项目目录命名习惯
- 不复用原项目接口路径与 DTO 结构
- 不复用原项目数据库表结构与权限实现
- 不把第三方示例代码直接纳入主线

设计原则：

- **先定义契约，再写实现**
- **模块优先，分层为模块内部实现**
- **框架适配在 transport 层收敛**
- **插件只扩展能力，不侵入核心**

# 2. 项目定位

## 2.1 项目名称

**GoAdmin**

## 2.2 产品定位

GoAdmin 是一个面向企业内部管理、权限控制、流程管理、业务扩展与平台化演进的后台管理系统基础平台。

- 用户、角色、菜单、权限管理

- 动态菜单与动态路由
- 插件扩展能力
- 多租户支持
- 前后端一体化开发体验

不对标以下内容：

- 原项目源码实现
- 原项目页面结构
- 原项目接口签名
- 原项目数据库表定义
- 原项目组件封装方式

# 3. 架构目标

## 3.1 设计目标

v1.1 的目标是建立一套可以长期演进的基础平台能力，而不是只做一个能跑的后台。

要达到的效果：

- 可以直接生成一个模块化单体工程
- 可以逐步增加业务模块而不破坏核心内核
- 可以按插件方式扩展路由、菜单、权限与事件订阅
- 可以先在 gin 适配层跑通完整链路，再扩展 go-zero / kratos
- 可以为后续微服务拆分保留边界

## 3.2 非目标

v1.1 暂不强求：

- 插件运行时热插拔与完全隔离
- 分布式事件总线的全链路可靠投递
- 多租户独立库的完整自动迁移工具
- 跨语言 SDK 的全自动生成
- 全量微服务拆分

## 3.3 核心升级方向

```text
模块化单体（Modular Monolith）
+ 插件化（Plugin System）
+ 事件驱动（Event Bus）
+ 多租户（Multi-Tenant）
+ 多框架适配（Gin / go-zero / Kratos）
+ 代码生成（CLI + AI + CodeGen）
+ DevOps 交付
```

# 4. 总体架构

## 4.1 架构视图

```text
Frontend (Vue3)
    ↓
Transport Layer (gin / go-zero / kratos)
    ↓
Application Layer
    ↓
Domain Layer
    ↓
Infrastructure Layer
```

## 4.2 关键原则

- **模块是第一公民**，分层是模块内部实现
- **业务逻辑不依赖具体框架**
- **框架差异只存在于 transport 层**
- **跨模块协作必须显式**，不能依赖隐式耦合
- **核心能力可被插件扩展，但不能被插件改写**

## 4.3 分层职责

### 4.3.1 Transport

负责：

- HTTP / RPC / MQ 入口适配
- 路由注册
- 请求参数绑定与校验
- 请求与响应格式转换
- 中间件挂载
- 框架上下文适配

### 4.3.2 Application

负责：

- 用例编排
- 事务边界
- 跨聚合协作
- 权限校验的应用层协调
- 事件发布
- DTO 与领域对象转换

### 4.3.3 Domain

负责：

- 实体、值对象、聚合根
- 领域规则
- 仓储接口
- 领域服务
- 领域事件定义

### 4.3.4 Infrastructure

负责：

- 数据库实现
- 缓存实现
- 消息队列实现
- 外部系统适配
- 仓储接口落地

# 5. 依赖规则

## 5.1 调用链与实现链

- **调用链**：`transport` → `application` → `domain`
- **实现链**：`infrastructure` → `domain`

## 5.2 强制约束

- `transport` 只能依赖 `application`
- `application` 只能依赖 `domain`
- `infrastructure` 可以实现 `domain` 定义的接口
- `domain` 不依赖具体框架、数据库、缓存、消息队列
- `handler` 不能直接访问 repository 实现
- `repository` 不能控制业务事务边界

## 5.3 跨模块协作原则

模块之间协作优先顺序：

1. `application service`
2. 领域事件 / EventBus
3. 内部契约接口
4. 未来可演进的 RPC / MQ

禁止：

- 直接引用别的模块 `infrastructure`
- 直接读写别的模块数据库模型
- 直接穿透到别的模块的内部包

# 7. 多框架适配设计

## 7.1 统一适配目标

GoAdmin 必须支持以下运行模式：

- `gin`：默认单体 HTTP 运行模式
- `go-zero`：面向微服务与高并发场景的适配模式
- `kratos`：面向强约束服务化场景的适配模式

三者必须共享同一套业务内核。

## 7.2 统一抽象

至少需要统一以下抽象：

- `Server`
- `Router`
- `Context`
- `Request`
- `Response`
- `Middleware`
- `Logger`
- `Config`

## 7.3 适配边界

所有框架差异只能存在于 `transport` 层：

- `application` 不感知 gin / go-zero / kratos
- `domain` 不感知 HTTP / RPC
- `infrastructure` 不依赖框架具体上下文
- 请求进入业务前必须转换为统一上下文

## 7.4 启动流程统一约定

1. 读取配置
2. 初始化日志、数据库、缓存、权限引擎
3. 装配模块与插件
4. 注册路由与中间件
5. 启动服务
6. 暴露健康检查与指标接口

# 9. 插件系统设计

## 9.1 插件目标

插件系统用于实现平台能力扩展，不用于侵入核心逻辑。

插件可以扩展：

- 路由
- 菜单
- 权限策略
- 事件订阅
- 定时任务
- 第三方集成

## 9.2 插件分级

```text
Core Plugin    → 平台级能力，如鉴权、权限、菜单、配置、审计
Business Plugin → 可独立演进的业务能力，如工单、审批、报表
External Plugin → 外部系统集成，如短信、IM、支付、对象存储
```

## 9.3 插件接口

```go
type Plugin interface {
    Name() string
    Version() string
    Metadata() PluginMetadata
    Init(ctx Context) error
    Register(registrar Registrar) error
    Start(ctx context.Context) error
    Stop(ctx context.Context) error
}
```

## 9.4 插件清单与发现

插件清单是插件进入系统的唯一可信描述，发现与注册必须解耦。

当前仓库已经具备的基线能力：

- 启动期显式加载插件
- 注册中心收敛路由、菜单、权限
- 插件元数据持久化与 CRUD 管理
- 启动时从注册中心种子同步到数据库

下一阶段应补齐：

- 本地目录扫描
- 清单文件解析
- 依赖与版本校验
- 兼容性校验
- 市场化包索引查询

每个插件必须具备清单或等价注册结构，至少包含：

- `package_id`
- `name`
- `version`
- `description`
- `enabled`
- `type`
- `source`
- `entrypoint`
- `dependencies`
- `capabilities`
- `menus`
- `permissions`
- `routes`
- `events`
- `checksum`
- `signature`
- `compatible_app_version`

发现来源建议分为四类：

- `builtin`：内置插件，随应用发布
- `local`：本地插件目录，适合开发与灰度
- `installed`：数据库中已安装插件的启用状态
- `marketplace`：插件市场目录或远程包仓库

发现流程建议固定为：

1. 扫描候选插件
2. 读取并解析清单
3. 校验清单 schema
4. 解析依赖图
5. 校验版本与应用兼容性
6. 生成加载计划
7. 交给运行时注册中心执行

## 9.5 生命周期与热插拔

插件生命周期应由独立的生命周期管理器统一编排，不能由业务 handler 直接驱动。

建议状态机为：

`discovered` → `installed` → `enabled` → `loaded` → `initialized` → `registered` → `running` → `stopping` → `stopped` → `disabled` / `failed`

生命周期动作建议定义为：

- `Discover`：发现并读取插件清单
- `Install`：落库、缓存包、写入元数据
- `Enable`：允许进入加载计划
- `Load`：装载代码或描述，建立运行上下文
- `Init`：注入配置、日志、容器、依赖
- `Register`：注册路由、菜单、权限、事件、定时任务
- `Start`：启动后台任务与订阅
- `Stop`：释放资源并优雅退出
- `Disable`：停止新流量并移出加载计划
- `Uninstall`：删除元数据与缓存包

热插拔要分级处理：

- **支持热切换**：无状态路由、菜单、权限、事件订阅变更
- **谨慎热切换**：涉及后台任务、缓存、外部连接池的插件
- **禁止直接热切换**：涉及数据库结构迁移、核心鉴权链路、运行时全局状态的插件

热切换推荐流程：

1. 标记目标插件为待切换
2. 停止对外接流或进入 drain 状态
3. 释放旧实例资源
4. 加载新版本插件
5. 重新注册运行时资源
6. 恢复对外服务
7. 若任一步失败，自动回滚到上一稳定版本

## 9.6 失败、隔离与回滚

插件系统必须默认“插件失败不拖垮主系统”，但不同级别插件应有不同处理策略。

建议策略：

- 核心平台插件失败：拒绝启动
- 业务插件失败：记录错误并跳过，不影响主程序
- 外部集成插件失败：允许降级，但必须告警
- 热更新失败：自动回滚到上一版本

隔离边界建议：

- 路由前缀必须命名空间化，避免冲突
- 菜单、权限、事件名称必须带插件前缀
- 插件内部数据表与迁移脚本必须独立
- 插件资源配置应限制超时、并发与连接数

## 9.7 插件市场化与分发

若要支持插件市场化，架构文档必须把“插件包”和“插件运行态”分开描述。

插件市场建议支持：

- 插件列表检索
- 版本查询
- 安装 / 升级 / 卸载 / 启用 / 禁用
- 版本锁定与回滚
- 包签名与校验
- 兼容性检查
- 审核状态与发布状态

插件市场元数据建议至少包含：

- `package_id`
- `name`
- `version`
- `publisher`
- `description`
- `category`
- `status`
- `download_url`
- `checksum`
- `signature`
- `compatible_app_version`
- `dependencies`
- `changelog`

对于市场插件，推荐的交付路径是：

1. 市场检索
2. 下载包并校验
3. 安装到本地插件缓存区
4. 解析清单并做兼容性检查
5. 注册到生命周期管理器
6. 启用并发布到运行态

## 9.8 插件目录演进建议

当前代码已经有：`interface/`、`loader/`、`registry/`、`application/`、`domain/`、`infrastructure/`、`transport/`、`builtin/`。

若要支持动态发现、热插拔、生命周期治理和插件市场化，建议进一步演进为：

```text
backend/plugin/
  manifest/        # 清单结构、schema、解析器
  discovery/       # 扫描、索引、依赖解析
  lifecycle/       # 状态机、启动/停止/切换编排
  marketplace/     # 市场目录、安装、升级、卸载
  runtime/         # 运行时装配与资源挂载
  loader/          # 插件装载
  registry/        # 运行时注册中心
  builtin/         # 内置插件
  application/
  domain/
  infrastructure/
  transport/
```

其中：

- `manifest/` 负责“插件是什么”
- `discovery/` 负责“插件在哪里”
- `lifecycle/` 负责“插件何时启停与切换”
- `marketplace/` 负责“插件如何安装与分发”
- `runtime/` 负责“插件如何进入运行态”

## 9.9 插件边界

允许：

- 新增 API
- 新增页面菜单
- 新增权限资源
- 订阅事件
- 接入第三方

禁止：

- 改写核心鉴权流程
- 改写基础错误码
- 直接修改核心数据结构
- 直接依赖内部实现细节
- 以插件方式绕过主系统的事务和审计约束

# 10. 事件驱动系统

## 10.1 设计目的

事件总线用于解耦模块与插件，适合：

- 状态变化通知
- 异步通知
- 插件订阅
- 跨模块弱耦合协作

## 10.2 边界约束

事件总线只负责通知，不负责复杂业务编排。

禁止：

- 把事件总线当事务总线
- 用事件绕过应用层事务边界
- 在事件处理器中堆叠大段核心业务流程

## 10.3 事件模型

```go
type Event interface {
    Name() string
    Source() string
    Payload() any
    Timestamp() time.Time
    TraceID() string
}
```

## 10.4 事件总线接口

```go
type EventBus interface {
    Publish(ctx context.Context, event Event) error
    Subscribe(name string, handler EventHandler) error
}
```

## 10.5 处理策略

- 默认按事件名分发
- 单个处理器失败不阻断其他处理器
- 关键事件需要重试与失败日志
- 本地实现优先采用内存队列或 channel
- 分布式实现后续再接入 Kafka / RocketMQ

## 10.6 示例事件

- `RoleAssignedEvent`

# 11. 多租户设计

## 11.1 设计目标

多租户能力必须在 v1.1 中预留，并支持运行时开关，至少要做到：

- 请求链路可识别租户
- 模型层可携带租户信息
- 查询层可显式过滤租户
- 权限策略可按租户隔离
- 支持通过配置关闭 tenant 能力，并在关闭后退化为单租户模式
- 关闭 tenant 后，写入操作忽略 `tenant_id`，查询操作不再注入租户过滤，鉴权退化为 role-only

## 11.2 tenant/ 目录落地结构

```text
core/
  tenant/
    context.go
    resolver.go
    middleware.go
    tenant.go
```

## 11.3 各文件职责说明

### 11.3.1 context.go

- 定义租户上下文的标准数据结构
- 在 tenant 启用时，提供 tenant 信息在请求链路中的读写能力
- 在 tenant 关闭时，相关读写应退化为 no-op，避免继续传播租户信息
- 负责将 `tenant_id`、`tenant_code`、`tenant_status` 统一封装为上下文对象
- 为 `application`、`domain`、`infrastructure` 提供统一读取入口

### 11.3.2 resolver.go

- 在 tenant 启用时负责从请求中解析租户信息
- 支持从 HTTP Header、JWT Claims、RPC Metadata、后台任务上下文中解析租户
- 解析结果应转换为统一的租户上下文对象
- 负责处理租户缺失、租户非法、租户禁用等基础校验
- 在 tenant 关闭时直接返回空租户，不再做租户校验或强制注入

### 11.3.3 middleware.go

- 负责将解析后的 tenant 信息注入请求上下文
- 在 HTTP / RPC 入口处自动执行租户识别与传递
- 保证后续业务层无需重复解析租户
- 可与鉴权中间件、日志中间件、审计中间件协同工作

### 11.3.4 tenant.go

- 定义租户核心轻量模型
- 作为平台级基础对象，不承载复杂业务逻辑
- 负责描述租户的关键字段、状态和基础行为约束
- 可作为查询过滤、策略判断、审计记录的共享数据模型

## 11.4 租户上下文约定

tenant 启用时，每个请求应携带：

- `tenant_id`
- `tenant_code`
- `tenant_status`

tenant 关闭时，上述字段可不提供，系统应按单租户模式运行。

注入来源：

- HTTP Header
- JWT Claims
- RPC Metadata
- 后台任务上下文

## 11.5 数据隔离模式

| 模式 | 说明 | 推荐程度 |
| --- | --- | --- |
| 共享库 + tenant_id | 所有租户共享数据库，按字段隔离 | 推荐 |
| 独立库 | 每个租户独立数据库 | 进阶 |

推荐规则：

- tenant 启用时默认采用共享库 + `tenant_id`
- tenant 启用时，所有租户相关查询必须显式带条件
- tenant 关闭时，查询层不得注入租户过滤，写入层不得保留传入的 `tenant_id`
- 全局资源必须声明为非租户资源
- 后续保留独立库演进空间

## 11.6 模型关系

```text
Tenant → User → Role → Policy
```

## 11.7 权限主体约定

tenant 启用时推荐主体格式：

```text
sub = tenant:user:role
obj = module:resource
act = create|read|update|delete|export|approve
```

tenant 关闭时，主体应退化为 role-only 语义，不再拼接 tenant 维度。

## 11.8 多租户请求流程

1. 请求进入 `middleware`
2. `resolver` 在 tenant 启用时从 Header / JWT / Metadata / 任务上下文中提取 tenant 信息
3. `context` 在 tenant 启用时统一封装租户对象并注入链路上下文
4. 业务层在 tenant 启用时读取 tenant 上下文，执行查询过滤与权限判断
5. `tenant` 信息在启用时参与审计、日志、权限与数据隔离决策；关闭时不再参与这些决策

## 11.9 强制要求

- 租户隔离逻辑不得散落在 controller 中
- 租户条件必须能在 repository 或查询构造阶段统一注入
- 租户状态变更必须记录审计日志
- `tenant/` 目录只负责上下文、解析、注入与轻量模型，不承载具体租户管理业务流程
- tenant 的启用/关闭语义必须在 context、resolver、service、repository、Casbin 各层保持一致

# 12. 权限与鉴权设计

## 12.1 鉴权方案

- 登录态采用 JWT，负责身份认证、登录态维护与 claims 传递
- 权限控制采用 Casbin，负责策略判定、角色继承与属性判断
- 授权模型支持 RBAC + ABAC，并在 tenant 启用时保留多租户隔离维度
- tenant 关闭时，鉴权应自动退化为 role-only 模式，不再依赖 tenant 上下文
- 关键操作必须写入审计日志，包含主体、资源、动作、结果与租户信息

## 12.2 Casbin 目录落地结构

```text
core/
  auth/
    casbin/
      model/
        rbac.conf
        abac.conf
      enforcer/
        enforcer.go
      adapter/
        adapter.go
        gorm_adapter.go
        file_adapter.go
      service/
        permission.go
        casbin_service.go
      watcher/
        redis_watcher.go
```

## 12.3 各目录职责说明

### 12.3.1 model/

- `rbac.conf`：RBAC 模型定义，描述 subject / object / action / policy / role 的基础关系
- `abac.conf`：ABAC 模型定义，描述属性表达式、上下文条件和租户维度约束
- 模型文件只定义规则，不承载业务代码

### 12.3.2 enforcer/

- `enforcer.go`：创建、初始化、缓存和复用 Casbin enforcer 实例
- 负责加载 model 与 adapter，形成可执行鉴权引擎
- 负责封装常用初始化逻辑，例如默认模型、默认策略、缓存策略与日志注入

### 12.3.3 adapter/

- `adapter.go`：定义统一的策略存储适配接口
- `gorm_adapter.go`：将策略持久化到数据库，支持策略表读写与批量同步
- `file_adapter.go`：将策略持久化到本地文件或静态策略源，适合本地开发与初始化场景
- 适配层只负责策略存取，不负责权限判断本身

### 12.3.4 service/

- `permission.go`：定义权限判断与策略管理的业务接口
- `casbin_service.go`：封装权限检查、策略增删改查、租户上下文注入与结果转换；tenant 关闭时自动退化为 role-only 评估
- service 是业务侧唯一应直接依赖的权限封装入口

### 12.3.5 watcher/

- `redis_watcher.go`：可选的分布式策略通知实现
- 用于多实例部署时同步策略变更、清理缓存或触发热更新
- 若暂不启用分布式刷新，可不落地该目录实现

## 12.4 权限对象约定

- `sub`：主体，tenant 启用时建议采用 `tenant:user:role` 结构；tenant 关闭时退化为 role-only 语义，必要时可扩展用户状态或组织信息
- `obj`：资源，建议统一为 `module:resource`，如 `user:list`、`menu:update`
- `act`：动作，建议统一为 `create|read|update|delete|export|approve`
- `env`：环境上下文，可由 JWT claims、请求头、租户上下文与业务上下文共同补充

## 12.5 权限运行流程

1. `middleware` 在 tenant 启用时从 JWT 中解析身份信息，并注入租户上下文
2. `service` 根据请求构造 `sub / obj / act / env`，tenant 关闭时只保留 role 维度
3. `enforcer` 载入对应模型并执行权限判断，tenant 关闭时使用 role-only 语义
4. `adapter` 提供策略读取、写入与同步能力
5. `watcher` 在多实例场景下通知其他节点刷新缓存
6. 结果写入审计日志，并转换为统一错误码或成功返回

## 12.6 权限能力

必须支持：

- 路由权限
- 菜单权限
- 按钮权限
- 数据权限预留
- 租户隔离权限
- 动态策略更新

## 12.7 权限运行时要求

- 权限决策可缓存
- 策略更新后可热更新
- 关键行为需要审计
- 权限错误必须返回统一错误码

## 12.8 与其他核心组件的协作

- `jwt` 负责认证，`casbin` 负责授权，两者职责分离
- `tenant` 负责在启用时提供租户上下文，关闭时退化为空；`casbin` 负责消费租户上下文，并在关闭时退化为 role-only
- `registry` 负责完成 `model`、`adapter`、`enforcer`、`service`、`watcher` 的装配与生命周期管理
- `event` 可在策略变更后发布权限刷新事件，供插件或其他模块订阅

- 所有核心接口应维护 OpenAPI 描述
- 生成器应支持从契约生成前端 API SDK
- 文档应与代码同步更新

# 14. CLI + AI + CodeGen 生成体系

## 14.1 目标

CLI、AI 与 CodeGen 用于降低重复性工作，提升模块开发效率，并形成统一的生成平台。

CodeGen 相关的详细架构与目录草案已拆分为独立设计文档：

- `docs/teckdesign/3. GoAdmin CodeGen 架构设计.md`
- `docs/teckdesign/3. GoAdmin CodeGen 目录结构草案.md`

主架构文档只定义 CodeGen 的定位、边界与协作方式，具体输入模型、生成流程、增量合并与多框架适配以独立文档为准。

CodeGen 的落地目录结构演进，请同步参考：

- `docs/teckdesign/3. GoAdmin CodeGen 目录结构草案.md`

## 14.2 CLI 命令示例

```bash
goadmin-cli generate module user
goadmin-cli generate crud order
goadmin-cli generate plugin demo
```

支持的常用参数：

- `--fields`：字段定义，格式为 `name:type,name2:type2`
- `--frontend`：同时生成前端 API、路由和视图文件
- `--policy`：生成并追加 Casbin policy
- `--force`：覆盖已存在的同路径文件

其中 `generate module` 只生成模块骨架与 `manifest.yaml`，`generate crud` 会生成 domain / application / infrastructure / transport 全栈代码，`generate plugin` 会生成插件运行时骨架并同步插件前端入口。

后续 CodeGen 将逐步从当前 `backend/cli/generate` 兼容层演进到 `backend/codegen` 核心层，以支持：

- DSL / DB 双驱动输入
- IR 统一中间模型
- 增量生成与手写保护
- gin / go-zero / kratos 适配器
- Vue3 / React 前端生成
- 低代码 UI 配置

## 14.3 生成输入

- 模块名
- 实体名
- 字段定义
- 主键与索引
- 是否生成前端页面
- 是否生成权限策略
- 是否生成测试模板

模块生成时，`module.go` 与 `manifest.yaml` 必须对齐现有模块约定：`name` 使用 snake_case，`kind` 默认是 `business-module`，`routes` 使用 `/api/v1/<plural>` 形式。

## 14.4 生成输出

- 模块目录骨架
- handler / service / model / repository
- 前端页面与路由
- Casbin policy
- 基础测试模板
- OpenAPI 草案

CodeGen 的产物必须满足：

- 默认不覆盖人工手写代码
- 支持 dry run / preview
- 支持二次生成与局部更新
- 生成结果应尽量可直接编译或只需少量调整即可编译

CRUD 生成结果应尽量贴近当前 `user`、`menu`、`role` 模块的字段风格：

- request 结构体使用 `json` / `form` tag
- response 结构体显式声明字段
- model 使用 `Clone()` 方法提供浅拷贝，并对 slice 字段做复制

## 14.5 生成器约束

- 生成内容必须符合当前架构约束
- 不得把框架细节写入 domain
- 支持二次生成与增量更新
- 生成结果应尽量可直接编译

## 14.6 AI 协同约束

统一提示词建议包含：

```text
【架构约束：GoAdmin v1.1】
```

AI 输出必须遵守：

- 先契约后实现
- 先模块后细节
- 先文档后代码
- 先内核后业务

## 14.7 CodeGen 架构约束

CodeGen 必须遵循以下约束：

- 解析层与生成层分离，DSL / DB 先统一到 IR
- 生成计划与模板渲染分离，不能把所有逻辑塞进单一模板目录
- 增量更新优先，手写区块必须受保护
- 运行时业务逻辑不进入 CodeGen
- 新框架、新页面类型、新插件输出必须通过适配器扩展，不得硬编码到主流程

CodeGen 的定位是平台生产力中枢，而不是运行时业务层。

# 17. 核心基础设施设计

## 17.1 Core 组件

```text
core/
  config/
  logger/
  errors/
  response/
  middleware/
  auth/
    jwt/
    casbin/
      model/
        rbac.conf
        abac.conf
      enforcer/
        enforcer.go
      adapter/
        adapter.go
        gorm_adapter.go
        file_adapter.go
      service/
        permission.go
        casbin_service.go
      watcher/
        redis_watcher.go
  event/
  registry/
  tenant/
  transport/
```

说明：

- `config/` 对应根目录 `config/`，负责配置加载与校验
- `registry/` 负责依赖注入、模块装配、插件注册与生命周期调度
- `transport/` 负责定义并收敛统一的路由、上下文与请求响应抽象

## 17.2 责任划分

- `config`：配置装载、合并、校验与环境隔离；其中 `auth.casbin.source` 用于选择权限策略来源，支持 `file` 与 `db`
- `logger`：结构化日志、trace 注入、脱敏与统一格式输出
- `errors`：统一错误码、错误包装与领域/接口错误转换
- `response`：统一响应格式与返回体封装
- `middleware`：鉴权、租户、日志、审计、恢复与跨域中间件
- `auth/jwt`：JWT 解析、签发、刷新与 claims 封装
- `auth/casbin/model`：RBAC / ABAC 模型定义与模型文件管理
- `auth/casbin/enforcer`：Casbin enforcer 初始化、缓存与实例管理
- `auth/casbin/adapter`：策略存储适配，支持 Gorm / File 等持久化方式；DB 模式会通过 Gorm 自动迁移 `casbin_model` 与 `casbin_rule`
- `auth/casbin/service`：权限判断、策略管理、业务封装与对外服务接口；会根据 `auth.casbin.source` 在文件模式与数据库模式之间切换，并在 DB 模式首次启动时用文件内容完成 seed
- `auth/casbin/watcher`：策略变更监听与分布式刷新（可选）
- `event`：事件总线抽象、本地发布订阅与异步派发
- `registry`：依赖注入容器、模块装配、插件注册与生命周期管理
- `tenant`：租户上下文注入、解析、传递与读取
- `transport`：统一 `Router` / `Context` / `Request` / `Response` 抽象定义

Casbin 的启动约定如下：

- `auth.casbin.source=file` 时，权限模型和策略均从本地文件加载
- `auth.casbin.source=db` 时，服务启动需要持有共享数据库连接，由 auth bootstrap 负责将 Casbin 初始化到与业务一致的数据库实例上
- DB 模式会自动创建 Casbin 相关表，并在首次启动时将 `rbac.conf` 和 `policy.csv` 的内容同步到数据库

## 17.3 核心协作方式

core 不承担具体业务实现，而是为模块化单体提供统一运行时能力：

- `registry` 负责把 `modules/`、`plugin/`、`transport/` 的初始化串联起来
- `transport` 负责屏蔽 gin / go-zero / kratos 的差异，向上提供稳定抽象
- `event` 负责模块与插件之间的解耦通知
- `tenant` 负责租户上下文在请求链路中的贯穿
- `auth` 负责认证与授权能力的基础封装，其中 `casbin` 目录承载模型、适配器、enforcer、服务层和可选 watcher 的完整权限实现

# 23. 交付验收标准

GoAdmin v1.1 的验收标准应满足：

- 代码结构符合 clean-room 约束
- 模块边界清晰
- transport / application / domain / infrastructure 分离明确
- 插件可扩展且不破坏核心
- gin / go-zero / kratos 可以共享同一业务内核
- 前端可按模块组织与扩展
- DevOps 能支撑本地开发、测试与生产部署
- 文档能直接指导后续开发

---

# 24. 结论

GoAdmin v1.1 的核心不是“复制一个后台系统”，而是构建一个 **可持续演进的平台内核**。

它的关键价值在于：

- 以模块化单体作为起点
- 以插件化保证扩展性
- 以事件驱动减少耦合
- 以多框架适配保证演进空间
- 以多租户预留支持企业化场景
- 以 DevOps 和生成器提升交付效率

**最终原则：模块自治、接口清晰、框架可替换、插件可扩展、文档先行。**
