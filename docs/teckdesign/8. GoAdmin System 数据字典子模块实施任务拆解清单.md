# GoAdmin System 数据字典子模块实施任务拆解清单

> **依据文档**：
> - `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`
> - `docs/teckdesign/7. GoAdmin System 数据字典子模块方案.md`
>
> **核心原则**：
> - 数据字典是 `System` 模块下的独立子模块，不是临时页面集合
> - 后端遵循 `transport -> application -> domain -> infrastructure` 分层
> - 前端复用现有 `Vue3 + TypeScript + Vite + Pinia + Vue Router + Axios + Element Plus` 体系
> - 严格保持 clean-room，不复用 Gin-Vue-Admin 的源码、接口签名、表结构与页面实现

---

# 1. 实施顺序建议

建议按以下顺序推进：

1. 建立模块骨架与 bootstrap 接入
2. 设计字典分类 / 字典项领域模型
3. 实现仓储与数据库迁移
4. 实现应用层用例编排
5. 实现 HTTP 接口与查询接口
6. 完成前端 System 菜单与页面
7. 补充权限、测试、审计与文档收尾

---

# 2. 任务拆解清单

## 任务 1：建立数据字典模块骨架与启动接入

### 目标
在 `server/modules/` 下新增 `dictionary` 模块骨架，并接入现有 bootstrap / migration / router 流程，使模块可以被系统启动识别。

### 主要工作
- 创建 `server/modules/dictionary/` 目录结构
- 增加 `module.go`、`bootstrap.go`、`manifest.yaml`
- 让模块实现与现有模块一致的注册方式
- 预留应用层、领域层、基础设施层、传输层目录
- 预留菜单挂载入口与系统路由入口

### 交付物
- 模块目录骨架
- 模块元信息文件
- bootstrap 注册代码
- 基础路由挂载入口

### 完成标准
- 新模块可以参与启动注册
- 迁移与路由入口符合当前模块约定
- 不依赖临时手工改 `main.go` 或核心路由文件

### 目标

- 定义分类与字典项之间的一对多关系

- 仓储接口定义
- 领域错误定义

- 模型能覆盖分类与项的核心字段
- 分类编码具备唯一性约束语义

# 3. 任务落地建议

建议按以下节奏推进：

- **第一步**：先完成模块骨架和领域模型
- **第二步**：再完成迁移与仓储
- **第三步**：然后补应用层和 HTTP 层
- **第四步**：最后接前端页面与权限收尾

这样可以避免一开始就把 UI、接口和数据库全绑死，降低返工成本。

---

# 4. 完成标准总览

数据字典子模块完成时，至少应满足：

- 系统菜单下可访问数据字典页面
- 后端具备分类和字典项的 CRUD 能力
- 支持按分类编码查询字典项
- 前后端分层清晰，没有跨层直接调用
- 权限与审计逻辑可扩展
- 文档、测试与实现保持一致

---

# 5. 结论

这份拆解清单的目标，是把“数据字典子模块”从概念方案变成可直接执行的任务序列。

首轮建议先做：

- 模块骨架
- 领域模型
- 数据库迁移

随后再推进：

- 应用层
- HTTP 层
- 前端页面
- 权限与测试收尾

这样最稳，也最符合当前 GoAdmin 的模块化单体演进路线。
