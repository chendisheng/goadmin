# GoAdmin CodeGen 目录结构草案 v1.0

> **定位**：GoAdmin CodeGen 的落地目录结构草案。
>
> **目标**：把 `3. GoAdmin CodeGen 架构设计.md` 中的能力拆成可以直接开发的目录边界、职责分工与演进顺序。
>
> **约束**：严格遵循 `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`，并与当前已有的 `server/cmd/cli`、`server/cli/generate` 实现保持兼容。

---

# 1. 草案目标

本草案只回答一个问题：**CodeGen 的代码应该怎么分目录，才能便于持续扩展、维护和落地实现。**

它不展开完整架构理论，只聚焦：

- 目录应该怎么拆
- 每个目录负责什么
- 哪些能力属于基础层
- 哪些能力属于未来演进层
- 当前实现如何平滑迁移

---

# 2. 设计原则

## 2.1 先稳定模型，再拆实现

CodeGen 最重要的不是先写模板，而是先稳定：

- 输入模型
- 中间模型 IR
- 生成计划
- 增量合并规则
- 目标框架适配边界

## 2.2 先保留现有能力，再补齐新边界

当前仓库已经具备：

- CLI 入口
- module / CRUD / plugin scaffold
- 字段解析
- 前端页面骨架生成
- Casbin policy 追加
- 基础模板渲染

因此目录草案必须以现有目录为起点，而不是推翻重建。

## 2.3 分层清晰，避免生成器变成“大泥球”

每个目录只能承担一种类型的职责：

- 解析输入
- 归一化模型
- 做生成计划
- 渲染模板
- 合并输出
- 写入文件
- 后处理

---

# 3. 推荐目录总览

```text
server/codegen/
  driver/          # CLI / HTTP / 任务入口
  schema/          # DSL / DB / manifest 输入解析
  model/           # 统一 IR 与资源模型
  planner/         # 生成计划、差异分析、冲突检测
  generator/       # 后端 / 前端 / 插件 / 策略生成
  merger/          # 增量合并、块级保护、手写区保留
  templates/       # 模板资源
  postprocess/     # 格式化、权限修补、路由补丁
  runtime/         # 生成任务运行环境与上下文
```

---

# 4. 各目录职责

## 4.1 `driver/`

职责：提供外部调用入口。

建议包含：

- `cli/`：命令行入口
- `http/`：远程生成接口
- `job/`：异步任务入口

适合放置：

- 参数解析
- 任务路由
- 生成请求接收
- dry run 入口

**注意**：`driver/` 只负责“接收请求”，不负责生成逻辑本身。

## 4.2 `schema/`

职责：读取并解析所有输入源。

建议包含：

- `dsl/`：YAML / JSON / TOML DSL 解析
- `database/`：数据库元数据读取
- `manifest/`：模块、插件、页面清单解析

这里解决的问题是：

- 输入到底长什么样
- 字段、关系、页面、权限如何表达
- 输入是否合法

## 4.3 `model/`

职责：定义统一中间模型 IR。

建议包含：

- `ir/`：统一资源模型
- `resource/`：模块、实体、字段、页面、插件等资源定义
- `diff/`：模型层差异结构

这里是 CodeGen 的核心抽象层，所有输入都应转换到这里再继续处理。

## 4.4 `planner/`

职责：决定“生成什么、跳过什么、更新什么”。

建议包含：

- 生成目标规划
- 冲突检测
- 生成范围裁剪
- 依赖顺序排序
- 预览与 dry run 输出

Planner 不写文件，只输出计划。

## 4.5 `generator/`

职责：按目标框架和目标端输出代码。

建议包含：

- `server/`
  - `gin/`
  - `gozero/`
  - `kratos/`
- `frontend/`
  - `vue3/`
  - `react/`
- `plugin/`
- `policy/`
- `config/`

这里负责模板渲染、适配器选择和代码生成。

## 4.6 `merger/`

职责：处理增量生成与手写保护。

建议包含：

- 块级更新
- 文件级保护
- 保留手写逻辑
- 局部替换
- 合并冲突处理

这是 CodeGen 能否真正用于生产的关键目录。

## 4.7 `templates/`

职责：统一存放模板资源。

建议按类型组织：

```text
templates/
  server/
  frontend/
  plugin/
  manifest/
  policy/
```

模板应版本化，避免未来升级破坏旧项目兼容性。

## 4.8 `postprocess/`

职责：对生成结果做收尾修正。

建议包含：

- 代码格式化
- 路由补丁
- 权限补丁
- 导入整理
- 命名统一

## 4.9 `runtime/`

职责：管理生成任务的运行上下文。

建议包含：

- 任务上下文
- 临时文件区
- 输出目录映射
- 生成日志
- 事务式回滚信息

# 5. 现有实现与目标目录的映射

## 5.1 当前目录

当前仓库已有：

```text
server/cmd/cli/
server/cli/generate/
```

## 5.2 当前实现职责

现有代码已经承担：

- CLI 启动
- module 生成
- CRUD 生成
- plugin 生成
- 字段解析
- 模板渲染
- Casbin policy 追加

## 5.3 建议迁移方式

建议分两步演进：

### 第一步：新增 `server/codegen/`

把新架构能力放入：

- `schema/`
- `model/`
- `planner/`
- `merger/`
- `generator/`
- `postprocess/`

### 第二步：让 `server/cli/generate/` 逐步收敛

把现有 CLI 目录降级为兼容层，只做：

- 参数解析
- 调用新 CodeGen 核心
- 保持旧命令可用

# 7. 分层职责边界

## 7.1 `schema` 只做解析

不做：

- 模板渲染
- 文件写入
- 路由补丁

## 7.2 `model` 只做统一表达

不做：

- 输入读取
- 输出写入
- 框架判断

## 7.3 `planner` 只做决策

不做：

- 文件写入
- 模板渲染
- 格式化

## 7.4 `generator` 只做输出

不做：

- 输入解析
- 差异判断
- 合并冲突解决

## 7.5 `merger` 只做合并

不做：

- 框架适配
- 业务逻辑
- UI 行为处理

## 7.6 `postprocess` 只做后置修正

不做：

- 生成计划
- 模型构建
- 输入解析

# 8. 目录演进顺序

## Phase 1：核心目录建模

先补齐：

- `schema/`
- `model/`
- `planner/`

## Phase 2：输出目录拆分

再补齐：

- `generator/server/`
- `generator/frontend/`
- `generator/plugin/`

## Phase 3：增量保护

再补齐：

- `merger/`
- `postprocess/`

## Phase 4：运行时化

最后补齐：

- `driver/http/`
- `runtime/`
- 异步任务支持

---

# 9. 与现有 CodeGen 实现的衔接建议

当前 `server/cli/generate` 中的能力建议按以下方式拆解：

- 字段解析 → `schema/`
- 资源结构 → `model/`
- 生成判断 → `planner/`
- 模板输出 → `generator/`
- 文件保护 → `merger/`
- 格式化与策略补丁 → `postprocess/`

这样可以避免后续继续往一个目录里堆模板和规则。

---

# 10. 目录草案总结

如果只保留一句话，这份目录草案的核心就是：

> **schema 管输入，model 管统一表达，planner 管生成决策，generator 管代码输出，merger 管增量保护，postprocess 管收尾修正。**

这套拆分方式更适合 GoAdmin 后续扩展到多框架、多前端、插件化和低代码场景。
