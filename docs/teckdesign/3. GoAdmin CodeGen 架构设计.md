# GoAdmin CodeGen 架构设计 v1.0（Clean-Room / Production Ready）

> **定位**：GoAdmin 的统一代码生成架构设计文档。
>
> **目标**：构建一个**可扩展 / 插件化 / 多框架支持 / 前后端解耦**的代码生成系统，用于模块、CRUD、插件、页面与配置资产的持续生成。
>
> **约束**：
>
> - 严格遵循 `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`
> - 保持 clean-room，不复用 Gin-Vue-Admin 的源码、模板、目录、接口、表结构或生成逻辑
> - 生成结果必须支持增量更新，**不能覆盖人工手写代码**
> - 支持 DSL / DB 双驱动
> - 支持 Vue3、React（可扩展）
> - 支持 gin、go-zero、kratos（可扩展）

---

# 1. 设计目标

## 1.1 核心目标

构建一个统一的生成平台，使 GoAdmin 能够从同一套元数据生成：

- 后端模块骨架
- 后端 CRUD 代码
- 后端插件骨架
- 前端页面、API、路由
- 权限策略
- 配置清单
- 低代码 UI 配置

## 1.2 必须满足

- 多后端框架支持
  - `gin`
  - `go-zero`
  - `kratos`
- 多前端支持
  - `Vue3`
  - `React`（可扩展）
- 插件化生成
- 增量生成，不覆盖手写代码
- DSL / DB 双驱动
- 低代码 UI 配置

## 1.3 生成器的职责边界

CodeGen 只负责“**从模型到代码**”的转换，不负责运行时业务逻辑。

CodeGen 不应直接承担：

- 鉴权判定
- 请求业务处理
- 数据业务规则
- 插件运行管理

这些职责应分别由核心运行时、插件系统和领域服务承担。

# 2. 设计原则

## 2.1 Clean-Room 原则

- 不沿用 Gin-Vue-Admin 的目录、模板、接口或结构体命名习惯
- 不复用其实现代码和生成规则
- 所有命名、路径、模板、交互方式均基于 GoAdmin 自身架构定义

## 2.2 单一事实来源

生成输入应来自两个可信源：

- DSL：适合设计态、版本化、可审阅
- DB：适合运行态、在线修改、低代码配置

生成输出必须围绕统一的中间模型（IR）展开，而不是直接从 UI 或数据库硬编码到模板。

## 2.3 增量优先

生成器默认不能覆盖用户手写内容。

必须支持：

- 文件存在检测
- 局部块更新
- 标记区块插入
- 可重放生成
- 变更冲突提示

## 2.4 框架适配器化

后端框架和前端框架都应通过 Adapter / Generator 接口抽象，避免模板逻辑和业务逻辑耦合。

# 3. 总体架构

## 3.1 运行链路

```text
输入来源
  ├── DSL 文件
  ├── 数据库元数据
  ├── CLI 参数
  └── 插件清单 / 市场资产
        ↓
Model Builder（统一中间模型 IR）
        ↓
Planner（生成计划 / 差异分析 / 冲突检测）
        ↓
Generator（按目标框架与目标端生成）
        ↓
Merger（增量合并 / 局部更新 / 保留手写区块）
        ↓
Writer（文件写入 / 事务式落盘）
        ↓
Post Actions（格式化 / 权限补丁 / 路由挂载 / 预览刷新）
```

## 3.2 分层结构

CodeGen 建议拆成六个子域：

- `schema`：DSL、数据库元数据、字段与关系定义
- `model`：统一中间模型 IR
- `planner`：生成计划、差异计算、冲突检测
- `generator`：后端、前端、插件、权限、配置生成
- `merger`：增量合并、区块保护、局部更新
- `driver`：CLI / API / 自动化调用入口

# 5. 核心概念

## 5.1 统一中间模型 IR

IR（Intermediate Representation）是 CodeGen 的核心。

所有输入最终都必须转成统一模型，再生成代码。

建议 IR 至少包含：

- `Module`
- `Resource`
- `Entity`
- `Field`
- `Relation`
- `Route`
- `Policy`
- `Page`
- `Component`
- `Plugin`
- `Manifest`
- `FrontendBinding`
- `BackendBinding`

## 5.2 生成目标类型

- `backend-module`
- `backend-crud`
- `backend-plugin`
- `frontend-page`
- `frontend-module-route`
- `policy`
- `manifest`
- `config`

## 5.3 生成方式

- **Full Generate**：首次生成整套脚手架
- **Incremental Generate**：对已有项目补充缺失资产
- **Patch Generate**：只改某些可控区块
- **Dry Run**：只输出差异，不落盘
- **Force Generate**：允许覆盖受控文件中的可重建区块

# 6. 输入模型设计

## 6.1 DSL 驱动

DSL 适合设计态输入，便于版本管理和审阅。

建议 DSL 支持：

- 模块定义
- 实体定义
- 字段定义
- 关系定义
- 前端页面定义
- 权限定义
- 插件定义
- 路由定义

### DSL 形态建议

可支持 YAML / JSON / TOML 之一，优先 YAML，便于人读和 diff。

示例语义：

```yaml
module: inventory
kind: business-module
framework:
  backend: gin
  frontend: vue3
entity:
  name: item
  fields:
    - name: id
      type: string
      primary: true
    - name: name
      type: string
      required: true
pages:
  - list
  - form
permissions:
  - inventory:view
  - inventory:edit
```

## 6.2 DB 驱动

DB 适合低代码和在线调整。

建议数据库至少存储：

- 资源定义
- 字段定义
- 关系定义
- 页面定义
- 组件配置
- 权限策略
- 生成版本
- 草稿状态
- 发布状态

DB 驱动适合：

- 低代码配置面板
- 运行时编辑
- 管理后台直接改配置
- 生成记录与版本回溯

## 6.3 双驱动策略

建议约定优先级：

- DSL 为主，DB 为辅：适合工程化、版本化项目
- DB 为主，DSL 为导出格式：适合低代码项目

最终由 `Model Builder` 将 DSL / DB 统一转换为 IR。

# 7. 生成流程设计

## 7.1 标准流程

1. 读取输入源（DSL / DB / CLI）
2. 归一化成 IR
3. 进行依赖与命名校验
4. 生成目标计划
5. 计算与现有文件的差异
6. 执行增量合并
7. 写入文件
8. 执行格式化与后置修补

## 7.2 关键校验

- 命名合法性
- 路由冲突
- 权限重复
- 页面路径冲突
- 模块依赖缺失
- 框架适配不兼容
- 手写区块冲突

## 7.3 生成计划

Planner 应输出：

- 要创建的文件
- 要更新的文件
- 要保护的区块
- 要跳过的内容
- 要告警的冲突

# 8. 增量生成与手写保护

## 8.1 核心要求

CodeGen 必须支持：

- 生成后再次运行不会无脑覆盖人工修改
- 支持局部区块替换
- 支持保留手写逻辑
- 支持模板重放

## 8.2 推荐机制

### 机制一：受控区块标记

文件中约定可覆盖区块，例如：

```text
// <goadmin:generated begin>
// 生成器管理的代码
// <goadmin:generated end>
```

生成器只更新该区块。

### 机制二：文件级保护

对人工文件或自定义扩展文件，默认只追加不覆盖。

### 机制三：差异合并

对模板文件先做 AST / 文本 diff，再合并变更。

## 8.3 覆盖策略

- 新文件：允许创建
- 受控模板文件：只覆盖标记区块
- 手写扩展文件：默认不覆盖
- 破坏性更新：必须显式加 `--force`

# 9. 后端框架适配设计

## 9.1 适配目标

必须支持以下后端框架：

- gin
- go-zero
- kratos

## 9.2 后端适配器职责

每个后端适配器负责：

- 路由组织
- Handler / Service / Repo 的代码形态
- 请求/响应模型风格
- 中间件挂载方式
- 配置与启动入口风格

## 9.3 统一生成约束

无论框架如何不同，CodeGen 都应保持统一的领域模型和资源描述，只在适配层输出框架特定代码。

## 9.4 框架输出差异

### gin

- HTTP 路由组
- 中间件链
- 直接 Handler 注册

### go-zero

- api spec / handler / logic / svc
- 分层更偏 RPC/HTTP 生成范式

### kratos

- transport / service / data 分离
- 生成 proto / service / repository 风格输出

# 11. 插件化生成设计

## 11.1 插件生成目标

CodeGen 本身也要支持插件化扩展。

可扩展点包括：

- 新后端框架适配器
- 新前端框架适配器
- 新模板包
- 新字段类型映射
- 新页面模板
- 新权限策略模板
- 新市场包导出器

## 11.2 插件接口

```go
type GeneratorPlugin interface {
    Name() string
    Version() string
    Apply(registry Registry) error
}
```

## 11.3 插件能力

插件可扩展：

- 模板注册
- 生成器注册
- 后处理器注册
- 规则校验器注册
- 转换器注册

## 11.4 生成器与插件系统协作

CodeGen 插件系统应与平台插件系统一致，但职责不同：

- 平台插件系统：扩展运行时能力
- CodeGen 插件系统：扩展代码生成能力

# 12. 模板与渲染策略

## 12.1 模板分层

建议模板分为三层：

- 基础模板：模块骨架、通用结构
- 框架模板：gin / go-zero / kratos / Vue3 / React
- 领域模板：CRUD / 插件 / 菜单 / 权限 / 页面

## 12.2 渲染要求

- 模板必须参数化
- 模板必须可测试
- 模板渲染后必须格式化
- 模板输出必须可追踪来源版本

## 12.3 版本化模板

模板建议带版本号，避免未来升级时破坏老项目兼容性。

# 13. 版本、迁移与兼容性

## 13.1 生成版本

每次生成建议记录：

- 输入来源版本
- 模板版本
- 框架版本
- 生成器版本
- 生成时间
- 生成结果摘要

## 13.2 兼容策略

- 老项目升级生成器时，默认只补齐缺失文件
- 破坏性修改必须显式确认
- 模板升级应支持迁移提示

## 13.3 回滚策略

生成前应保留快照或变更记录，以便：

- 回退单文件
- 回退单资源
- 回退整次生成

# 16. 与低代码 UI 的协作

  ↓

## 16.3 适合低代码的对象

# 17. 与插件系统的协作

CodeGen 与插件系统应该形成互补关系：

- 插件系统定义“运行时怎么扩展”
- CodeGen 定义“扩展代码怎么生成”

建议 CodeGen 支持：

- 生成插件 manifest
- 生成插件后端骨架
- 生成插件前端页面
- 生成插件权限与菜单
- 生成插件市场元数据

这可以直接服务于未来的插件市场和动态插件体系。

# 19. 安全与质量控制

## 19.1 安全控制

- 不执行未授权覆盖
- 不在生产环境直接执行破坏性生成
- 不接受不兼容 DSL
- 不生成未校验市场包内容

## 19.2 质量控制

- 模板单测
- 生成结果快照测试
- 差异测试
- 框架适配测试
- 插件扩展测试

## 19.3 可观测性

每次 CodeGen 运行应记录：

- 输入源
- 生成对象
- 创建/更新/跳过数量
- 冲突项
- 错误项
- 生成耗时

# 20. 现有实现与差距

当前仓库中的生成器已经具备：

- CLI 入口
- module / CRUD / plugin scaffold
- 字段解析
- 前端页面生成
- Casbin policy 追加
- 基础模板渲染

当前差距主要在于：

- 还没有统一 IR
- 还没有 Planner / Merger
- 还没有 DSL / DB 双驱动
- 还没有多框架后端适配
- 还没有 React 前端适配
- 还没有低代码 UI Schema
- 还没有生成历史与回滚

因此，当前实现适合作为第一阶段基础，后续按本架构继续演进。

# 22. 结论

GoAdmin CodeGen 的正确方向不是“模板拼接工具”，而是一个**统一模型驱动、框架可扩展、前后端解耦、支持增量生成、支持 DSL / DB 双驱动、支持低代码 UI 的生产级生成平台**。

如果按本文档实施，CodeGen 可以成为 GoAdmin 平台的基础生产力中枢，并持续服务于模块开发、插件开发和低代码交付。
