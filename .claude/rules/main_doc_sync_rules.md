# GoAdmin Main Doc Sync Rules

## 1. Source of Truth

- 主架构来源始终是 `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`。
- 若新增了独立设计文档，例如插件系统、CodeGen、DevOps、前端专题等，必须先判断它是否影响主架构文档的定位、边界、目录、阶段计划或引用关系。
- 任何独立文档都不能替代主架构文档；它只能作为主架构的补充与展开。

## 2. Sync Rules

- 当独立设计文档新增、拆分或重命名时，要同步检查主架构文档是否需要增加引用说明。
- 当独立设计文档覆盖某个主文档章节的细节时，主文档应保留“定位 / 边界 / 关系 / 引用”，不要在主文档重复堆叠实现细节。
- 主文档负责定义平台级原则、目录骨架、实施阶段和模块边界；独立文档负责定义具体方案、目录拆分、接口和流程。
- 如果某项能力已拆出独立文档，主文档必须明确指向该文档，避免后续设计和实现脱节。
- 若主文档与独立文档出现表述冲突，应优先核对仓库真实实现，再决定修正文档或实现。

## 3. CodeGen-Specific Rules

- `docs/teckdesign/3. GoAdmin CodeGen 架构设计.md` 与 `docs/teckdesign/3. GoAdmin CodeGen 目录结构草案.md` 必须同时被视为 CodeGen 的配套文档。
- 主架构文档中涉及 CodeGen 的部分，只保留定位、边界、阶段计划和引用，不要把 IR、Planner、Merger、模板、适配器等细节全部展开。
- 当 CodeGen 目录结构发生变化时，主架构文档中的 `backend/codegen/` 描述、阶段计划和引用说明必须同步更新。

## 4. Do / Don't

### Do

- 保持主文档简洁、稳定、可作为导航入口。
- 保持独立文档详细、可落地、可实施。
- 保持引用关系清晰，避免重复描述。
- 保持主文档和独立文档的阶段计划一致。

### Don't

- 不要让主文档变成实现细节堆栈。
- 不要让独立文档脱离主文档的架构约束。
- 不要新增独立文档而忘记补主文档引用。
- 不要让 CodeGen 设计绕开 clean-room 和分层规则。
