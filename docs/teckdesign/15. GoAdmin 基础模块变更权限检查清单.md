# GoAdmin 基础模块变更权限检查清单

> **适用范围**：`auth`、`user`、`role`、`menu`、`tenant`、`audit`、`system`、`plugin`、`codegen` 等基础模块。
>
> **使用目的**：当基础模块新增、修改、删除接口、菜单、权限、默认策略或页面映射时，避免遗漏 Casbin policy、菜单权限、默认种子和文档同步。
>
> **依据文档**：
> - `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`
> - `docs/teckdesign/13. GoAdmin CodeGen 业务模块删除设计方案.md`
> - `docs/teckdesign/14. GoAdmin CodeGen 业务模块删除实施任务拆解清单.md`
>
> **核心原则**：
> - 基础模块的接口变更，必须同步权限资产，不得只改代码不改授权
> - `backend/core/auth/casbin/adapter/policy.csv` 是 Casbin policy 的源头，`db` 模式必须自动跟随 `csv` 的新增、删除和修改
> - `file` / `db` 两种 Casbin 存储方式都要覆盖
> - 路由、菜单、policy、默认种子、前端映射必须保持一致
> - 删除接口时，必须同时删除失效权限与失效菜单，避免遗留可访问入口

---

# 1. 变更前检查

## 1.1 先确认变更类型

- [ ] 新增接口
- [ ] 删除接口
- [ ] 修改接口路径
- [ ] 修改接口方法（GET / POST / PUT / DELETE）
- [ ] 新增菜单
- [ ] 删除菜单
- [ ] 修改菜单 permission
- [ ] 修改默认授权策略
- [ ] 修改模块 manifest
- [ ] 修改前端页面映射
- [ ] 修改 DB seed / 迁移逻辑

## 1.2 先确认影响范围

- [ ] 受影响模块名称已明确
- [ ] 受影响接口路径已列出
- [ ] 受影响菜单项已列出
- [ ] 受影响权限对象 `obj` 已列出
- [ ] 受影响动作 `act` 已列出
- [ ] 是否存在 `file` 与 `db` 两种 Casbin 存储方式已确认
- [ ] 是否依赖默认种子规则已确认

## 1.3 先做全局检索

- [ ] 检索目标路径是否已存在于 `backend/core/auth/casbin/adapter/policy.csv`
- [ ] 检索目标路径是否已存在于 `backend/core/auth/casbin/enforcer/enforcer.go`
- [ ] 检索目标路径是否已存在于模块 `manifest.yaml`
- [ ] 检索目标路径是否已存在于前端路由或页面映射
- [ ] 检索目标 permission 是否已存在于菜单默认数据

---

# 2. 接口变更检查

## 2.1 新增接口时

- [ ] 为新接口补充 Casbin policy
- [ ] 如果使用默认种子，补充 `defaultRules()`
- [ ] 如果使用文件策略，补充 `policy.csv`
- [ ] 如果使用数据库策略，确认启动同步或回填逻辑会自动跟随 `policy.csv` 的新增规则
- [ ] 为新接口确认 `obj` 与 `act` 命名是否合理
- [ ] 为新接口补充或更新测试用例

## 2.2 删除接口时

- [ ] 删除对应 Casbin policy
- [ ] 删除默认种子中的失效规则
- [ ] 删除菜单或页面中的失效入口
- [ ] 删除模块 manifest 中失效路由声明
- [ ] 删除前端路由或按钮权限中的失效引用
- [ ] 确认历史 DB policy 不会继续放行失效接口

## 2.3 修改接口路径或方法时

- [ ] 更新对应 policy 的 `obj`
- [ ] 更新对应 policy 的 `act`
- [ ] 更新菜单 permission 或页面权限绑定
- [ ] 更新相关文档中的路径示例
- [ ] 更新自动化测试中的请求路径

---

# 3. 权限数据检查

## 3.1 Casbin policy 检查

- [ ] `policy.csv` 已同步更新
- [ ] `defaultRules()` 已同步更新
- [ ] DB seed / 启动同步逻辑与 `policy.csv` 保持一致
- [ ] DB 现有策略已自动补齐或回填完成
- [ ] 重复 policy 已去重
- [ ] 无关 policy 未被误删

## 3.2 路由权限检查

- [ ] 所有新增 API 都有对应授权数据
- [ ] 所有删除 API 都已移除授权数据
- [ ] `GET / POST / PUT / DELETE` 的动作映射正确
- [ ] `RequirePermission` 实际校验的路径与 policy 路径一致

## 3.3 菜单权限检查

- [ ] 菜单 `permission` 与 API 授权保持一致
- [ ] 菜单可见性与接口可访问性一致
- [ ] 目录菜单与叶子菜单的权限语义正确
- [ ] 菜单删除时不会遗留孤儿路由入口

## 3.4 默认授权检查

- [ ] 新环境首次启动后不会因为缺少默认策略而 403
- [ ] `file` 模式和 `db` 模式的默认权限表现一致
- [ ] 初始化后管理员账号能访问基础模块管理入口
- [ ] 非管理员账号不会因为默认种子被误放行

---

# 4. 模块清单与页面映射检查

## 4.1 manifest 检查

- [ ] 模块 `manifest.yaml` 已更新
- [ ] `routes` 已同步新增或删除
- [ ] `menus` 已同步新增或删除
- [ ] `permissions` 已同步新增或删除
- [ ] `capabilities` 与实际能力一致

## 4.2 前端映射检查

- [ ] 动态路由映射已更新
- [ ] 页面组件路径已更新
- [ ] 菜单与页面名称保持一致
- [ ] 若为基础模块新增页面，前端路由和侧边栏能正常展示
- [ ] 若为基础模块删除页面，前端不会再出现无效入口

---

# 5. 数据库与启动检查

## 5.1 `file` 模式

- [ ] `backend/core/auth/casbin/adapter/policy.csv` 已更新
- [ ] 本地启动后可直接生效
- [ ] 重启后策略仍然有效
- [ ] `policy.csv` 中的规则是最终来源，文件内容更新后无需手工改 DB

## 5.2 `db` 模式

- [ ] 启动时会自动从 `policy.csv` 同步缺失规则到 `casbin_rule`
- [ ] 已存在的 `casbin_rule` 与 `casbin_model` 数据不会和新规则冲突
- [ ] 新增规则会自动跟随 `policy.csv` 同步到 DB，无需手工单独维护两份权限
- [ ] 如需删除失效规则，已提供迁移脚本或一次性修复方案
- [ ] 生产环境切换前已验证策略同步结果

---

# 6. 测试检查

## 6.1 最小回归测试

- [ ] 新增接口的正向鉴权测试通过
- [ ] 删除接口的失效策略测试通过
- [ ] admin 账号能访问基础模块管理接口
- [ ] 非授权角色被正确拒绝
- [ ] `file` 模式测试通过
- [ ] `db` 模式测试通过

## 6.2 必须覆盖的回归点

- [ ] 路径变更不会漏改 policy
- [ ] 删除接口不会残留可访问 policy
- [ ] 默认种子不会漏掉新接口
- [ ] 菜单权限不会与 API 权限脱节

---

# 7. 审计与发布检查

- [ ] 涉及权限变更的内容已记录到变更说明
- [ ] 涉及删除权限的内容已同步记录风险点
- [ ] 需要的话已补充审计日志或操作说明
- [ ] 文档已同步更新
- [ ] 发布前已复核“路由、菜单、policy、默认种子”四项一致性

---

# 8. 推荐执行顺序

1. 先改接口与路由
2. 再改菜单与 manifest
3. 再改 Casbin policy 和默认种子
4. 再补 DB seed / 回填
5. 再补测试
6. 最后更新文档与发布说明

---

# 9. 一句话原则

**凡是基础模块接口变更，都必须同步检查“路由、菜单、policy、默认种子、前端映射”是否一致。**
