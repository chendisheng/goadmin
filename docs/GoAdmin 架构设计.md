# GoAdmin 总体架构设计（正式版）

> **定位**：面向企业级后台管理平台的 clean-room 架构设计文档。
>
> **核心方向**：模块化单体 + 插件化扩展 + 事件驱动 + 多租户预留 + 多框架适配 + 代码生成 + DevOps 交付。
>
> **说明**：本文是 GoAdmin 架构设计正式版；另有单独的简版摘要文件用于快速浏览。
>
> **适用范围**：本文档描述 GoAdmin 当前主线的总体架构、目录规范、运行时编排、扩展方式与阶段性演进路线，作为项目公开后的单一架构说明。

---

# 1. 文档目标

GoAdmin 的目标不是单纯做一个 CRUD 管理后台，而是构建一个可以长期演进的平台内核。这个内核需要同时满足：

- 可独立运行
- 可按模块扩展
- 可按插件扩展
- 可生成代码和资源
- 可支撑多租户演进
- 可支持前后端分离部署
- 可支持持续集成与容器化交付

本文档希望回答四个问题：

- **系统是什么**：GoAdmin 的产品边界和能力边界是什么
- **系统怎么分层**：后端、前端、插件、生成器、部署体系如何划分
- **系统怎么运行**：启动顺序、依赖初始化、模块注册、权限加载如何完成
- **系统怎么演进**：新模块、新插件、新页面、新生成器如何接入而不破坏主干

---

# 2. 项目定位

## 2.1 产品定义

GoAdmin 是一个面向企业内部管理场景的后台管理平台基础设施，提供用户、角色、菜单、权限、插件、上传、代码生成、国际化、部署交付等通用能力。

它的设计重点不是堆积功能，而是建立一套稳定的能力边界，使后续业务模块可以通过统一契约快速接入。

## 2.2 设计目标

- **平台化**：系统本身是一个平台内核，而不是一个孤立业务系统
- **模块化**：每个业务能力都以独立模块组织
- **可扩展**：模块、插件、生成器、菜单、权限都可扩展
- **可交付**：支持 Docker、Compose、Kubernetes、Helm、CI/CD
- **可维护**：明确依赖方向，减少跨层耦合
- **可公开**：文档与代码表达统一，避免隐式依赖

## 2.3 不追求的目标

当前架构不强求：

- 运行时插件热插拔的完全隔离
- 分布式事件总线的全链路可靠投递
- 多租户独立库的自动迁移闭环
- 跨语言 SDK 的全自动生成
- 一开始就拆分为微服务

这些能力可以作为后续演进方向，但不应破坏当前模块化单体的主架构。

# 3. 设计原则

## 3.1 Clean-Room 原则

GoAdmin 只参考公开可见的通用工程范式，不复用外部项目源码、目录惯性、接口签名、页面实现或数据库结构。

必须遵守：

- 不复制现成源码
- 不沿用外部项目的目录组织习惯
- 不复用既有接口路径和 DTO 结构
- 不复用既有表结构与权限实现
- 不把第三方示例代码直接纳入主线

## 3.2 模块优先

GoAdmin 不是“按层写一堆目录”，而是“按模块组织能力”。

原则是：

- 模块是第一公民
- 分层是模块内部实现
- 模块之间通过显式契约协作
- 共享能力沉淀到 core、contract 或 shared 级别

## 3.3 分层解耦

后端采用统一的四层结构：

- `transport`
- `application`
- `domain`
- `infrastructure`

其中：

- `transport` 负责协议适配
- `application` 负责用例编排
- `domain` 负责核心业务规则
- `infrastructure` 负责基础设施落地

依赖必须单向流动，不能反向污染。

## 3.4 插件优先

系统能力必须允许通过插件方式扩展：

- 路由
- 菜单
- 权限
- 事件订阅
- 定时任务
- 第三方集成

插件只能扩展能力，不能改写核心契约。

## 3.5 事件驱动

跨模块协作优先通过事件完成，而不是直接互相调用内部实现。

适合事件化的场景包括：

- 用户创建后同步初始化数据
- 权限或菜单变更后的缓存刷新
- 资源导入导出后的通知处理
- 插件注册后的元信息同步

## 3.6 生成优先

重复性工作尽量由生成器完成：

- 模块骨架
- CRUD 代码
- 插件骨架
- 菜单和权限资源
- 国际化资源骨架
- 删除与清理能力的辅助资源

生成器不是附属工具，而是平台能力的一部分。

## 3.7 安全默认

所有能力默认遵循安全优先策略：

- 认证和鉴权默认开启
- 私有资源默认走鉴权访问
- 权限与菜单必须显式授予
- 删除与变更必须提供预览和审计
- 配置项必须支持生产环境安全覆盖

# 4. 总体架构

## 4.1 架构视图

```text
浏览器 / 前端应用
    ↓
HTTP 入口层（transport）
    ↓
应用层（application）
    ↓
领域层（domain）
    ↓
基础设施层（infrastructure）
```

在平台维度上，还存在几个横切能力：

```text
core 运行时
  ├─ 配置
  ├─ 日志
  ├─ 事件总线
  ├─ 认证/授权
  ├─ 国际化
  ├─ 多租户
  └─ 启动编排

modules 业务模块
  ├─ user
  ├─ role
  ├─ menu
  ├─ auth
  ├─ dictionary
  ├─ upload
  ├─ casbin
  └─ 其他生成模块

plugin 插件系统
  ├─ builtin 插件
  ├─ 插件定义
  ├─ 插件菜单
  ├─ 插件权限
  └─ 插件页面/路由

codegen 代码生成体系
  ├─ CLI
  ├─ DSL
  ├─ 预览
  ├─ 生成
  └─ 删除/清理
```

## 4.2 核心架构判断

GoAdmin 当前采用的是 **模块化单体**，不是微服务。

原因是：

- 当前阶段更需要统一认证、统一权限、统一菜单、统一生成器
- 模块化单体更适合快速交付与持续演进
- 后续如果需要拆分，可以从模块边界自然演进出去

## 4.3 关键不变量

无论后续如何扩展，以下不变量必须保持：

- **请求入口统一走 transport 层**
- **业务规则不直接依赖框架实现**
- **模块之间不通过内部包强耦合**
- **插件只接触约定接口，不接触内部实现细节**
- **数据库结构与代码生成资源必须保持一致**

# 5. 仓库与目录规范

## 5.1 顶层目录职责

```text
.
├── server/       后端工作区
├── web/          前端工程
├── deploy/       容器化与集群交付资源
├── docs/         架构、设计、需求、参考资料
├── memory-bank/  项目状态与过程记录
├── Makefile      根级入口命令
└── README.md     英文项目说明
```

## 5.2 后端目录规范

### `server/core/`

存放平台级核心能力：

- 配置
- 日志
- 事件总线
- 认证与授权
- 国际化
- 多租户
- 注册与编排
- 统一错误与响应
- 传输抽象

### `server/modules/`

存放业务模块。每个模块按统一结构组织：

```text
module/
  application/
  domain/
  infrastructure/
  transport/
  bootstrap.go
  module.go
  manifest.yaml
```

### `server/plugin/`

存放插件系统相关实现：

- 插件定义
- 插件加载器
- 插件注册上下文
- 内置插件
- 插件持久化与元数据

### `server/codegen/`

存放代码生成、安装、删除、DSL 执行与资源生成相关实现。

### `server/cmd/`

存放服务端入口：

- `cmd/server`：主服务入口
- `cmd/cli`：命令行入口
- `cmd/migrate`：迁移或辅助工具入口

## 5.3 前端目录规范

`web/` 是独立前端工程，采用 Vue 3 + TypeScript + Vite + Pinia + Vue Router + Axios + Element Plus。

主要目录：

- `web/src/api/`：API 客户端
- `web/src/auth/`：认证与会话
- `web/src/components/`：公共组件
- `web/src/router/`：路由与动态菜单
- `web/src/store/`：Pinia 状态管理
- `web/src/views/`：页面视图
- `web/src/i18n/`：国际化运行时与资源

## 5.4 交付目录规范

`deploy/` 用于存放可交付产物和部署模板：

- Dockerfile
- Compose
- Kubernetes manifests
- Helm chart
- Nginx 配置
- CI/CD 配置

# 6. 运行时启动流程

## 6.1 启动顺序

后端主服务的启动顺序应保持稳定：

1. 解析项目根目录
2. 加载配置
3. 初始化日志
4. 打开数据库连接
5. 初始化鉴权运行时
6. 初始化认证服务
7. 初始化事件总线
8. 注册事件订阅
9. 迁移内置模块
10. 迁移生成模块
11. 初始化菜单与插件持久化数据
12. 安装模块 manifest 到运行时菜单/权限体系
13. 加载插件
14. 启动 HTTP 服务
15. 监听退出信号并优雅关闭

## 6.2 运行时依赖

核心依赖至少包括：

- `Config`
- `Logger`
- `DB`
- `EventBus`
- `AuthorizationRuntime`
- `MenuService`
- `PluginService`
- `ProjectRoot`

这些依赖应通过显式依赖注入传递，避免隐式全局变量。

## 6.3 配置加载原则

配置按“默认值 + 文件 + 环境变量”合并：

- 默认值提供可运行基线
- 文件提供环境差异化
- 环境变量用于容器和 CI/CD 覆盖

当前配置目录约定为：

- `server/config/config.yaml`
- `server/config/config.dev.yaml`
- `server/config/config.prod.yaml`

## 6.4 启动失败原则

任何关键依赖初始化失败都应立即终止启动，而不是带病运行。

典型失败包括：

- 数据库无法连接
- 鉴权运行时不可用
- 插件元数据加载失败
- 模块迁移失败
- 菜单或 manifest 安装失败

# 7. 后端核心运行时

## 7.1 core 职责

`core` 是平台级基础能力集合，负责为所有模块提供统一的运行环境。

主要子能力：

- `config`：配置加载与默认值
- `logger`：结构化日志
- `event`：本地事件总线
- `auth`：认证、授权、令牌、政策运行时
- `bootstrap`：模块注册与迁移协调
- `i18n`：国际化资源加载与语言选择
- `tenant`：多租户上下文与开关
- `transport`：统一的路由注册契约
- `registry`：运行时对象容器
- `errors`：统一错误码与错误包装

## 7.2 认证与授权

认证与授权是平台的核心边界之一。

当前架构支持：

- JWT 登录态
- 用户身份加载
- 角色与菜单权限映射
- Casbin 风格的策略运行时
- 文件或数据库作为策略来源
- 运行时重载策略
- 默认策略种子化

## 7.3 国际化

国际化属于平台级能力，不应只停留在前端。

当前设计应支持：

- 系统级语言
- 模块级语言
- 插件级语言
- 用户级语言偏好
- 资源自动加载
- 语言回退

资源加载应覆盖：

- core locales
- modules locales
- plugin locales
- codegen locales

## 7.4 多租户

多租户当前以预留和统一上下文传递为主，核心要求是：

- 默认情况下所有可写数据都带租户上下文
- 查询链路可按租户过滤
- 关闭租户开关时系统可退化为单租户模式
- 业务代码不应自行拼接租户逻辑

# 8. 模块化单体设计

## 8.1 模块定义

模块是 GoAdmin 的基本交付单元。每个模块都应该包含：

- 领域模型
- 应用服务
- 基础设施实现
- HTTP 入口
- bootstrap 适配
- manifest 描述

## 8.2 模块契约

模块应暴露统一契约：

- 模块名
- manifest 路径
- 迁移逻辑
- 路由注册逻辑

统一契约使得平台可以对模块做：

- 自动迁移
- 自动注册
- manifest 安装
- 权限/菜单同步
- 代码生成与删除

## 8.3 模块内部结构

标准模块结构如下：

```text
server/modules/<module>/
  application/
  domain/
  infrastructure/
  transport/
  bootstrap.go
  module.go
  manifest.yaml
```

如果模块还需要前端资源、国际化资源或额外脚本，也应在模块内建立清晰边界，不应散落到全局目录中。

## 8.4 模块边界

模块之间协作时应遵循：

- 通过接口或事件协作
- 禁止直接依赖对方内部实现
- 禁止跨模块复用非契约对象
- 公共逻辑应抽到 core 或共享层

## 8.5 manifest 作用

manifest 是模块对外声明能力的方式，至少描述：

- 模块名称
- 路由
- 菜单
- 权限
- 能力标签
- 扩展说明

manifest 的目标是让“模块定义”和“运行时安装”保持一致。

# 9. 事件驱动系统

## 9.1 设计目的

事件总线用于解耦跨模块操作，避免模块之间相互调用对方内部服务。

## 9.2 事件总线范围

当前以本地事件总线为主，适合：

- 用户创建后自动触发后续处理
- 菜单/权限变更后的刷新通知
- 插件加载后的同步动作
- 生成器安装/删除流程中的异步辅助操作

## 9.3 事件设计原则

- 事件名稳定、可读
- 事件载荷轻量
- 事件处理器幂等
- 事件消费失败要可追踪

## 9.4 适用边界

事件总线不承担：

- 事务替代
- 强一致跨库协调
- 大规模分布式消息可靠投递

这些是后续演进方向，不应混入当前主架构。

---

# 10. 插件系统设计

## 10.1 插件目标

插件系统用于在不修改核心的前提下扩展平台能力。

## 10.2 插件能力

插件可以提供：

- 路由
- 菜单
- 权限
- 页面
- 任务
- 事件订阅
- 第三方集成

## 10.3 插件加载方式

当前采用静态加载 + 元数据持久化的模式：

- 内置插件先加载
- 插件注册到运行时容器
- 插件定义落库
- 插件菜单和权限同步到数据库

## 10.4 插件边界

插件不得：

- 直接修改核心鉴权流程
- 直接改写基础响应格式
- 直接变更核心模块数据结构
- 绕过平台约定的注册方式

## 10.5 插件页面约定

插件页面应通过统一的动态路由和资源映射机制接入前端，而不是手工硬编码到主路由树中。

# 11. 权限与鉴权设计

## 11.1 认证

认证以 JWT 为核心，登录成功后返回：

- 访问令牌
- 刷新令牌（如启用）
- 用户信息
- 角色信息
- 权限信息
- 语言偏好

## 11.2 授权

授权以资源 + 操作 + 主体为基础，兼容菜单和接口两类场景：

- API 路径权限
- 菜单可见权限
- 按钮级权限
- 插件权限

## 11.3 策略来源

支持两种策略来源：

- 文件
- 数据库

数据库模式下，启动时应构建内存运行时并通过 seed/reload 完成策略同步，避免不必要的启动写入。

## 11.4 默认权限模型

默认权限模型应满足：

- 公开健康检查可访问
- 登录前接口显式放行
- 其他业务接口默认受控
- 管理员具备默认完整权限

## 11.5 权限和菜单关系

菜单决定入口是否可见，权限决定接口是否可调用。

两者必须保持一致，否则会出现：

- 有菜单无权限
- 有权限无菜单
- 按钮可见但接口被拒绝

因此菜单、权限和路由必须在同一套 manifest / 资源同步机制下维护。

---

# 12. API 与数据契约

## 12.1 API 风格

GoAdmin 的 API 以 REST 风格为主，路径与资源语义应保持稳定和可读。

## 12.2 响应规范

统一响应应包含：

- code
- message
- data
- request_id（如有）
- localized message（如有）

## 12.3 数据契约原则

- 请求结构和响应结构分离
- DTO 不直接暴露领域实体内部状态
- API 参数命名应明确、稳定
- 不依赖前端页面行为来决定后端数据结构

## 12.4 分页与列表

列表类接口应统一分页、排序、过滤约定，以便前端管理页复用。

# 13. 代码生成与 CLI 体系

## 13.1 目标

代码生成体系用于减少重复工作，并保持项目结构一致。

## 13.2 支持范围

当前生成体系应覆盖：

- 模块生成
- CRUD 生成
- 插件生成
- DSL 执行
- 预览与生成
- 删除与清理
- 资源骨架生成

## 13.3 生成原则

- 先预览，再执行
- 生成内容要可回滚、可追踪
- 生成的代码必须符合当前仓库约定
- 生成器输出的资源应与运行时契约一致

## 13.4 生成内容

典型生成内容包括：

- 后端应用层、领域层、基础设施层、传输层
- 前端 API、路由、视图、资源文件
- 国际化资源骨架
- Casbin 策略资源
- schema 和 manifest 相关文件

## 13.5 删除与清理

删除能力与生成能力成对设计：

- 删除前必须可预览
- 删除必须识别归属边界
- 删除应区分自有资源和共享资源
- 删除后应验证运行时剩余资源的完整性

---

# 14. 前端架构设计

## 14.1 前端目标

前端的目标不是单纯渲染页面，而是作为平台交互层，统一承担：

- 登录与会话
- 路由与菜单
- 权限与按钮控制
- 管理页表单与列表
- 国际化
- 插件页面接入
- 运行时状态管理

## 14.2 技术栈

前端采用：

- Vue 3
- TypeScript
- Vite
- Pinia
- Vue Router
- Axios
- Element Plus
- i18next 生态

## 14.3 前端分层

建议分为：

- `api`：请求封装
- `auth`：认证与会话
- `store`：状态管理
- `router`：路由与动态菜单
- `components`：通用组件
- `views`：页面视图
- `i18n`：国际化
- `utils`：公共工具

## 14.4 动态路由与菜单

前端菜单和路由应来自后端资源模型，而不是纯前端静态写死。

这样可以实现：

- 角色不同，菜单不同
- 插件接入后自动出现入口
- 生成模块接入后自动挂载页面

## 14.5 KeepAlive 与页面刷新

管理页通常挂在 KeepAlive 场景下，因此列表页、筛选页、详情页需要考虑：

- 首次进入加载数据
- 切回缓存页重新刷新数据
- 语言切换后文案刷新
- 菜单切换后状态重建

## 14.6 权限指令

前端应提供统一权限指令或组件封装，用于按钮级控制。

按钮是否显示只是 UI 约束，后端接口权限仍然必须二次校验。

---

# 15. 上传与资源访问设计

## 15.1 上传模型

上传能力采用“元数据 + 存储驱动 + 访问链路”的模式。

至少区分：

- 文件元数据
- 存储位置
- 访问可见性
- 内容类型
- 文件大小
- 校验信息

## 15.2 存储驱动

支持不同存储实现：

- 本地文件系统
- 数据库存储
- 对象存储

统一对外暴露：

- 上传
- 读取
- 删除
- 是否存在
- 公网访问地址
- 签名访问地址

## 15.3 访问原则

- 私有资源默认走鉴权下载
- 公开资源可以直接访问
- 前端预览优先使用安全链路
- 不应假设 `public_url` 一定是浏览器可直达地址

---

# 16. 数据设计与迁移规范

## 16.1 表设计原则

- 统一命名风格
- 主键、索引、唯一约束明确
- 列类型与索引长度匹配
- 表设计服务于运行时和生成器，而不是临时页面

## 16.2 迁移原则

- 内置模块可自动迁移
- 生成模块可按配置自动迁移
- 迁移必须与模块生命周期绑定
- 迁移失败应明确暴露，不应静默忽略

## 16.3 资源表与业务表分离

平台级资源表与业务数据表应明确分离，例如：

- 用户、角色、菜单、权限
- 插件定义
- 模块 manifest 资源
- 上传元数据
- 策略与模型数据

## 16.4 兼容性原则

表结构变更时要优先考虑：

- 线上兼容
- 生成器兼容
- 权限数据兼容
- 迁移可回滚性

# 17. DevOps 与交付设计

## 17.1 本地开发

本地开发必须支持两种方式：

- Docker Compose
- 宿主机直接开发

宿主机开发应配合本地缓存目录，避免污染全局环境。

## 17.2 容器化交付

仓库应提供：

- 后端镜像构建
- 前端镜像构建
- Compose 本地编排
- Nginx 静态部署

## 17.3 集群交付

仓库应同时支持：

- Kubernetes manifests
- Kustomize overlays
- Helm chart

## 17.4 CI/CD

CI/CD 应覆盖：

- 单元测试
- 类型检查
- 前端构建
- 后端构建
- 镜像构建与发布
- 部署流水线

## 17.5 健康检查

服务应暴露可用于部署探活的健康检查接口，并在容器与编排层复用。

# 18. 安全与审计

## 18.1 安全目标

- 默认鉴权
- 默认最小权限
- 关键操作可审计
- 删除与批量变更可追踪
- 私有文件不可被绕过访问

## 18.2 审计重点

建议审计以下动作：

- 登录与退出
- 角色与权限变更
- 菜单与路由变更
- 插件启用/禁用
- 上传与删除
- 生成与删除操作
- 策略刷新与重载

## 18.3 错误处理

错误输出应满足：

- 对外信息可读
- 对内便于排查
- 不泄露敏感细节
- 支持国际化

# 19. 分阶段实施路线

## Phase 1：Core

目标：建立配置、日志、响应、错误、启动骨架。

验收标准：

- 服务可启动
- 配置可加载
- 健康检查可访问

## Phase 2：鉴权与权限

目标：接入 JWT、授权运行时、菜单权限基础模型。

验收标准：

- 可登录
- 可鉴权
- 可基于权限控制菜单与接口

## Phase 3：基础模块

目标：完善用户、角色、菜单、字典、上传等基础模块。

验收标准：

- 基础管理页可用
- 列表、编辑、删除、绑定流程可跑通

## Phase 4：插件系统

目标：支持插件定义、菜单、权限、页面接入。

验收标准：

- 插件页面可加载
- 插件菜单可显示
- 插件可独立扩展 UI

## Phase 5：CodeGen / CLI

目标：支持模块、CRUD、插件生成与删除能力。

验收标准：

- 可预览
- 可生成
- 可删除
- 生成资源与运行时一致

## Phase 6：多租户与事件驱动

目标：将租户上下文和事件总线纳入主干。

验收标准：

- 租户字段可传递
- 事件可订阅
- 跨模块协作更清晰

## Phase 7：前端完善

目标：增强动态路由、国际化、可维护组件和页面体验。

验收标准：

- 路由、菜单、权限一致
- 国际化生效
- 页面切换体验稳定

## Phase 8：Build & DevOps

目标：完成前端部署、镜像构建、集群交付和 CI/CD。

验收标准：

- build 成功
- 静态资源可访问
- 可通过 Nginx 提供服务
- 支持生产环境部署

建议按以下顺序推进：

2. Core 四件套
3. 鉴权与权限
4. 基础模块

# 21. 结论

GoAdmin 的核心不是“复制一个后台系统”，而是构建一个 **可持续演进的平台内核**。

它应当做到：

- 在结构上保持 clean-room
- 在实现上保持模块化单体
- 在扩展上支持插件和生成器
- 在运行上具备统一的认证、权限、国际化与事件能力
- 在交付上支持现代化 DevOps 流程

只要这些边界保持清晰，GoAdmin 就可以在不破坏主干的前提下持续扩展业务能力。
