# GoAdmin CodeGen 一键生成并安装实施任务拆解清单

> **依据文档**：
> - `docs/teckdesign/1.GoAdmin 架构设计 v1.1.md`
> - `docs/teckdesign/3. GoAdmin CodeGen 架构设计.md`
> - `docs/teckdesign/6. GoAdmin CodeGen 生成后自动安装到系统方案.md`
> - `docs/teckdesign/13. GoAdmin CodeGen 业务模块删除设计方案.md`
> - `docs/teckdesign/16. GoAdmin CodeGen 一键生成并安装设计方案.md`
>
> **核心原则**：
> - 一键流程以 `apply = generate + install + refresh` 为目标，不再让用户手工拆成两步
> - `policy.csv` 是 Casbin policy 的源头，DB 模式必须自动跟随 `csv` 变化
> - `apply` 与现有 `delete` 链路必须形成生命周期配对，但不是严格数学意义上的无损逆操作
> - 先预览，后执行；先编排，后接入；先后端核心，后 CLI / HTTP / UI
> - 严格保持 clean-room，不复用 Gin-Vue-Admin 的源码、接口签名、目录结构与实现逻辑

---

# 1. 实施顺序建议

建议按照下面顺序推进：

1. 定义 `apply` 的请求、结果与编排契约
2. 实现生成 + 安装的后端应用编排层
3. 接入 `manifest` 安装、菜单写入与 policy 跟随
4. 接入 CLI 一键命令
5. 接入 HTTP API 与 UI 操作入口
6. 补齐验证、审计、幂等与失败恢复
7. 完成测试、文档与收尾

---

# 2. 任务拆解清单

## 任务 1：定义 `apply` 的请求、结果与编排契约

### 目标
建立一键生成并安装所需的最小数据契约，让 CLI、HTTP API、UI 与内部编排服务能够共用同一套输入输出结构。

### 主要工作
- 定义 `ApplyRequest`、`ApplyPlan`、`ApplyResult` 等核心结构
- 统一模块名、生成类型、`force`、`dry-run`、`install`、`refresh` 等参数语义
- 统一生成摘要、安装摘要、policy 摘要、冲突与告警的输出字段
- 明确 `apply` 与 `delete` 的配对关系和职责边界
- 定义旧模块或旧生成物的兼容处理策略

### 交付物
- `backend/codegen/model/lifecycle` 中的 `apply` 请求 / 计划 / 结果数据结构
- 统一的执行状态枚举或结果类型
- 输入输出字段说明
- 最小兼容规则说明

### 完成标准
- `apply` 的输入输出能够覆盖生成、安装、同步、验证四个阶段
- CLI / HTTP / UI 可以共享同一套契约
- 契约不绑定具体前端框架或具体交互实现

## 任务 2：实现后端 `apply` 应用编排层

### 目标
建立一个统一的后端编排服务，负责把生成、安装、policy 跟随和验证串成单次执行流程。

### 主要工作
- 在 `backend/codegen/application/apply` 中封装生成器调用，复用现有 `GenerateModule`、`GenerateCRUD`、`GeneratePlugin`、`GenerateManifest` 与 `AppendPolicyLines`
- 封装安装器调用，复用现有 manifest 安装服务
- 串接 policy 追加、缓存刷新与结果聚合
- 支持 dry-run 预览与真实执行两种模式
- 处理执行中间失败的错误归类与状态回收

### 交付物
- `backend/codegen/application/apply` 编排服务
- 生成与安装的统一调用入口
- 执行结果聚合逻辑
- 失败分类与日志/审计基础能力

### 完成标准
- 一次调用即可完成生成与安装
- dry-run 不产生写入副作用
- 生成失败或安装失败时能清晰返回阶段性错误
- 编排层不直接承担模板渲染或菜单业务逻辑

### Prompt 提示词
```text
你现在要实现 GoAdmin CodeGen 一键生成并安装方案中的「任务 2：实现后端 apply 应用编排层」。

请严格遵循以下约束：
1. `apply` 必须复用现有生成器和安装器，不要重写模板系统。
2. 编排层只负责组织流程，不承担业务规则和 UI 展示逻辑。
3. 预览模式必须只读，不能写文件、不能写数据库。
4. 执行结果要能区分生成失败、安装失败、policy 同步失败、验证失败。
5. 保持 clean-room，不复用外部项目的编排方式。

本任务需要你：
- 实现 apply 编排服务
- 串接生成与安装
- 输出统一结果
- 增加必要测试

请输出：
- 代码实现
- 调用流程说明
- 失败场景说明
- 测试建议
```

## 任务 3：接入 `manifest` 安装、菜单写入与 policy 跟随

### 目标
让一键流程在生成代码后自动把 `manifest.yaml` 接入系统运行态，并保证 Casbin policy 按“CSV 为源、DB 自动跟随”的规则同步。

### 主要工作
- 调用 `manifest` 安装服务完成菜单 upsert
- 确保菜单、路由、权限安装与生成结果保持一致
- 在生成阶段同步追加 `policy.csv`
- 在 DB 模式下确保启动同步或安装后同步能把新增规则写入 DB
- 补齐安装后验证与缓存刷新逻辑

### 交付物
- `manifest` 安装链路接入
- 菜单写入 / 更新逻辑
- policy 跟随与刷新逻辑
- 安装后验证逻辑

### 完成标准
- 生成后的模块可以直接进入系统菜单体系
- 新增 policy 可以在 file / db 两种模式下保持一致
- 安装过程不会覆盖人工手工维护的其他数据
- 安装结果具备可审计性与可追踪性

# 3. 实施完成标准

## 3.1 功能完成标准

- `apply` 可以一次完成生成 + 安装 + 刷新
- `delete` 可以作为反向生命周期能力执行清理
- CLI / HTTP / UI 三种入口都能复用同一套编排逻辑
- `policy.csv` 与 DB policy 跟随规则保持一致

## 3.2 工程完成标准

- 重复执行不产生重复脏数据
- 失败可定位到阶段
- 共享资源不会被误删
- 人工资产不会被默认覆盖

## 3.3 文档完成标准

- `apply` 的职责边界清晰
- 与 `delete` 的配对关系清晰
- 团队可以按任务清单直接进入开发

---

# 4. 收尾建议

建议实际推进顺序如下：

1. 先做 `apply` 的契约与编排服务
2. 再接 CLI 命令
3. 再接 HTTP API / UI
4. 再补验证与审计
5. 最后补测试与文档同步

---

# 5. 结论

一键 `generate + install` 的最小开发路径是可行的，推荐以 `apply` 作为统一编排命令：

- `apply` = 生成 + 安装 + 刷新
- `delete` = 计划 + 清理 + 验证

这样既能直接进入开发，也能和现有删除链路形成稳定的生命周期闭环。
